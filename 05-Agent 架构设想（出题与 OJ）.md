# GOC 项目 Agent 架构设想（出题与 OJ）

当前常见做法是用 **Agents / Teams** 来规划与执行工作：多个角色分工、按流水线或任务树协作，而不是单一大模型「一口气全干」。下面是一种可能的 Agent 架构，对应「出题 + OJ MVP」的拆分方式，便于你理解或落地到 Cursor / 多 Agent 平台。

---

## 一、整体思路

- **规划层**：一个「规划 / 协调」角色，负责拆任务、定顺序、验收标准，不直接写代码或出题。
- **执行层**：若干**领域 Agent**（出题、Schema/规范、执行器、前端、判题），各吃一份输入、产出一份交付物。
- **衔接**：规划 Agent 产出「任务卡」或「工单」（含输入文档、产出物定义、验收条件）；执行 Agent 按工单执行，完成后可交给**验收 Agent** 或规划 Agent 做检查。

可以是「多 Agent 各跑一轮、人工或调度器传手」，也可以是「单会话里用提示词切换角色」的轻量实现。

---

## 二、角色划分（Agents / Teams）

| 角色 | 职责 | 输入 | 产出 | 典型动作 |
|------|------|------|------|----------|
| **规划 / 协调 Agent** | 拆解《05-实施计划》与《出题与 OJ 一次性完成》为可执行工单；定依赖顺序；汇总验收 | 05 实施计划、出题+OJ 范围文档 | 任务列表（含步骤、依赖、验收条件）、里程碑 | 读 05/06/07；输出「Step 2 工单：写 07，依据 06+03」；检查交付清单 |
| **需求 / 规范 Agent** | 把「题目长什么样、系统要做什么」固化成 Schema 与规范文档 | 06 案例与模式、03 判题/题库、01 指令 | 07 Schema、08 分类与难度、10 轨迹规范（或合并进 07） | 写 07/08；保证与 06、01、03 一致 |
| **出题专家 Agent** | 按 07/08/09 生成题目，保证题型多样、难度分层、含 solution | 06 模式、07 Schema、08 分类、09 出题指南、01 指令 | problems/*.json（含 description、solution、tags、difficulty），不包含轨迹 | 批量生成 10～20 道题；不照抄 06 表述 |
| **执行器 Agent** | 实现 GOC 解析与执行，输出标准轨迹 JSON | 01 指令、10 轨迹规范（或 07 中轨迹部分） | goc-executor.js（浏览器可跑）+ 轨迹格式说明 | 实现 parser + 执行循环；对 sample solution 跑通并输出轨迹 |
| **轨迹与判题 Agent** | 用执行器给每道题生成 expectedTrajectory；实现判题逻辑（轨迹比对 + 容差） | problems/*.json（有 solution）、执行器、容差规则 | 更新题目 JSON（写入 expectedTrajectory）；judge.js（浏览器内比对） | 批跑 solution → 写回 JSON；实现 compare(expected, actual) |
| **前端 / OJ Agent** | 实现单页 OJ：题目列表、做题页、编辑器、运行、提交、结果展示 | 02 页面结构、07 题目格式、index.json、执行器与判题接口约定 | 静态前端（HTML/JS 或 React/Vue）：列表页、做题页、Canvas 绘图、localStorage 记录 | 读 index + 题目 JSON；接执行器与判题；不依赖后端 |
| **验收 Agent** | 按交付清单逐项检查：文件存在性、格式、端到端流程 | 05 交付清单、07 Schema、出题+OJ 文档中的验收表 | 验收报告（每项 PASS/FAIL + 路径或说明） | 检查 06～10 文档、problems/、oj/ 目录、运行一道题全流程 |

## 三、协作流程（流水线式）

```
[规划 Agent]
  读 05 + 出题与OJ一次性完成
  → 输出：工单列表（Step 2～6 + OJ MVP 4 块）、依赖关系、验收标准

[需求/规范 Agent]
  接工单：写 07 / 08 / 10（或 07 含轨迹）
  输入：06, 01, 03
  → 产出：07, 08, 10（或 07 扩展）

[出题专家 Agent]
  接工单：生成题目
  输入：06, 07, 08, 09, 01
  → 产出：problems/*.json（无 expectedTrajectory）+ index.json

[执行器 Agent]
  接工单：实现 GOC 执行器
  输入：01, 10（或 07 中轨迹）
  → 产出：goc-executor.js + 轨迹格式文档

[轨迹与判题 Agent]
  接工单：生成轨迹 + 判题逻辑
  输入：problems/*.json（solution）、执行器、容差规则
  → 产出：更新后的 problems/*.json（含 expectedTrajectory）、judge.js

[前端/OJ Agent]
  接工单：单页 OJ
  输入：02, 07, index.json, 执行器, judge
  → 产出：oj/ 或 web/ 下静态页面与资源

[验收 Agent]
  接工单：全量验收
  输入：交付清单、仓库当前状态
  → 产出：验收报告
```

依赖关系简要：**07/08 先于出题**；**执行器先于轨迹与判题**；**轨迹 + 题目 JSON 先于前端**。规划 Agent 可据此排出顺序（例如：规范 → 出题 → 执行器 → 轨迹与判题 → 前端 → 验收）。

## 四、和你现在做法的对应关系

- **单 Agent（当前）**：一个 Claude 会话 + 一份「出题 + OJ 一次性执行」提示词，相当于把规划、规范、出题、执行器、轨迹、前端、验收都压在一个角色里，按提示词里的顺序自己做。
- **多 Agent / Teams**：把上述 7 个角色拆开，每个角色只做一块；由「规划 Agent」或你本人发工单、传手。适合在 Cursor Composer 里分多次会话（每次一个角色）、或在多 Agent 平台上配置 7 个 Agent，由调度器按依赖顺序触发。

若你有多 Agent 或 Teams 产品（如 Cursor 多会话、AutoGen、CrewAI、LangGraph 等），可以按上表配置 7 个角色与输入输出；若暂时只有单会话，仍用「一个提示词包办」即可，只是心里知道逻辑上可以拆成这 7 块。

---

## 五、简化版（3 个 Agent）

若不想拆太细，可以合并为 3 个「团队」：

| 团队 | 合并的角色 | 产出 |
|------|------------|------|
| **规范与出题** | 规划 + 需求/规范 + 出题专家 | 07/08/09/10、problems/*.json + index（无轨迹） |
| **引擎与判题** | 执行器 + 轨迹与判题 | goc-executor.js、题目 JSON 补全 expectedTrajectory、judge.js |
| **前端与验收** | 前端/OJ + 验收 | 静态 OJ 页面、验收报告 |

顺序：规范与出题 → 引擎与判题 → 前端与验收；规划可以只做一次（你或一个轻量规划提示词），输出「三步工单」即可。

---

## 六、小结

- **Agent 架构**：规划 + 需求/规范 + 出题 + 执行器 + 轨迹与判题 + 前端/OJ + 验收，共 7 角色；可简化为 3 个团队。
- **协作方式**：规划拆工单、定顺序与验收 → 各执行 Agent 按工单产出 → 验收 Agent 按清单检查。
- **和当前提示词的关系**：现有「出题 + OJ 一次性执行」提示词 = 单 Agent 串起上述全部步骤；若改用 Agents/Teams，就按本文拆成多角色、多会话或多次调用即可。
