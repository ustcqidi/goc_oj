# 出题 + OJ 一次性完成：范围与提示词

「把出题和 OJ 都干完」可以做到，但**完整 OJ** 和 **OJ MVP** 工作量差很多。本文档界定「一晚可完成」的 MVP 范围，并给出一次性执行的提示词。

---

## 一、OJ 完整版 vs OJ MVP

| 维度 | 完整 OJ（03 文档愿景） | OJ MVP（一晚可交付） |
|------|------------------------|----------------------|
| **前端** | 编辑器 + 绘图区 + 命令参考 + 题目列表 + 提交历史 + 用户中心 + 多页路由 | 单页：题目列表 + 题目页（编辑器 + 绘图区 + 运行/提交）+ 命令参考区 |
| **执行引擎** | GOC 解析 → AST → 轨迹，支持 step、cin/cout | 浏览器内 JS 实现：解析 GOC → 执行得到轨迹（与 01 指令一致） |
| **判题** | 轨迹比对 + 图形比对 + 结构检查，服务端判题 | **仅轨迹比对**，在浏览器内完成（expectedTrajectory 从题目 JSON 来） |
| **题库** | 后端 API + 数据库 | **静态**：读取 `problems/index.json` + 各题 JSON，无后端 |
| **用户与记录** | 登录、提交记录存库、进度统计、多角色 | **无登录**；做题记录用 **localStorage** 存（题目 ID + 通过/未通过 + 最后代码） |
| **部署** | 前后端 + DB + 判题服务 | **纯静态**：前端 HTML/JS/CSS + 若干 JSON，可 `file://` 或任意静态托管 |

MVP 的目标：**孩子能选题 → 写代码 → 运行看图 → 提交 → 立刻看到对/错**，无需你搭服务器、不用登录，题目和判题都在本地/静态环境完成。

## 二、一次性做完的内容清单（出题 + OJ MVP）

### 阶段一（出题）— 不变

1. 06 题目案例与模式（已有则跳过）
2. 07 题目数据格式（Schema）
3. 08 题目分类与难度体系
4. 09 出题专家规范
5. `problems/` 下 10～20 道题（JSON）+ 每题的 **expectedTrajectory**（由执行引擎跑 solution 生成，见下）
6. `problems/index.json` 索引

### 阶段二 MVP（OJ）

7. **执行轨迹规范**：定义轨迹 JSON 格式（cmd、args、顺序等），与 01 指令一一对应，便于执行器输出与判题比对。
8. **GOC 执行器（浏览器 JS）**：能解析并执行 01 规定的 GOC 子集（pen.fd/rt/lt/bk、pen.c/size/up/down、for/if、变量等），输出轨迹数组；不要求支持全部指令，先覆盖现有题目用到的。
9. **题目预生成轨迹**：对 `problems/` 里每道题的 solution 跑一遍执行器，把得到的轨迹写入该题 JSON 的 `expectedTrajectory` 字段（可写脚本用 Node 跑同一套执行逻辑，或手填几道后其余用执行器批量生成）。
10. **前端单页应用**：
    - 题目列表页：从 `problems/index.json` 读入，展示题目标题、难度、标签；点进某题进入做题页。
    - 做题页：左侧或上方题目描述，右侧代码编辑器（含语法高亮或至少等宽），运行按钮（执行器跑当前代码 → 用轨迹在 Canvas 上绘图），提交按钮（当前代码跑出轨迹，与本题 expectedTrajectory 比对，显示 Pass/Fail）。
    - 可选：命令参考区（01 常用指令）、清屏/画坐标、localStorage 记录每题最后提交结果与代码。
11. **判题逻辑（浏览器内）**：比对两条轨迹（顺序一致、指令类型一致、数值在容差内），返回 Pass/Fail；容差规则可写死（如坐标 ±2、角度 ±1°）。

技术选型建议（由 Claude 自定）：纯 HTML + 原生 JS，或 React/Vue 单页；执行器用 JS 实现；无后端、无构建也可（若用 React 可 Vite 一把梭）。部署方式：静态文件可放 `goc/oj/` 或 `goc/web/`，用 `index.html` 入口。

## 四、提示词：出题 + OJ MVP 一次性执行

下面整段复制给 Claude，用于在**无需你再输入**的前提下，完成「阶段一 6 步 + 阶段二 MVP 上述 4 项」。

```
你是 GOC 「出题 + OJ MVP」的全程执行者，要在**不向我提问**的前提下，完成以下全部工作。

【第一部分：阶段一（出题）】
- 若 06/07/08/09 或 problems/ 下题目与 index 已存在，则跳过或只补缺；否则按《05-实施计划（先出题后 OJ）.md》完成 Step 2～6。
- 产出：07/08/09 文档、problems/ 下至少 10 道题的 JSON、problems/index.json。每道题的 JSON 里需要留出 expectedTrajectory 字段（可先空数组，第二部分再填）。

【第二部分：阶段二 OJ MVP】
- 范围：**纯前端、无后端、无登录**。做题记录用 localStorage；题目与判题均在浏览器内完成。
- 你必须完成：
  1. **执行轨迹规范**：写一份简短的轨迹 JSON 规范（与 01 指令对应，如 `{ "cmd": "fd", "args": [100] }`），可放在 07 末尾或单独 10-执行轨迹规范.md。
  2. **GOC 执行器（JS）**：在浏览器可运行的 JS，能解析并执行 01 规定的 GOC 子集（至少 fd/rt/lt/bk、c/size、up/down、for、变量），输出轨迹数组。不要求支持 01 全部指令，先覆盖当前题库用到的。
  3. **题目 expectedTrajectory**：用上述执行器对每道题的 solution 跑一遍，把输出轨迹写入该题 JSON 的 expectedTrajectory 字段。
  4. **前端单页 OJ**：
     - 题目列表：读取 problems/index.json，展示题号、标题、难度、标签；点击进入做题页。
     - 做题页：题目描述、代码编辑器、运行（执行器执行当前代码 → 用轨迹在 Canvas 上绘图）、提交（执行器执行当前代码得到轨迹，与本题 expectedTrajectory 比对，显示 Pass/Fail）。可选：命令参考、清屏、localStorage 存每题最后一次提交结果与代码。
  5. **判题逻辑**：在浏览器内比较两条轨迹（指令顺序与类型一致，数值在容差内），容差可写死（如坐标 ±2、角度 ±1°）。

【技术选型】
- 由你决定：纯 HTML+JS 或 React/Vue 等；执行器与判题逻辑可放在单独 .js 文件中。所有内容放在可静态托管的目录（如 goc/oj/ 或 goc/web/），不依赖后端服务。

【约束】
- 不向我提问；遇到歧义时自行做合理选择并保持前后一致。
- 先保证「能跑通」：选一道简单题（如画正方形）从列表 → 做题 → 运行 → 提交 → 看到对错，再扩展其余题目与体验。

【验收】
- 全部完成后，按本文档「三、交付物汇总」逐项自检，并输出验收报告（每项已交付/未交付 + 路径或说明）。
```

## 五、若只做「出题」或只做「OJ」时

- **只做出题**：用《05-实施计划-Claude自主执行提示词.md》里的「提示词 A」或最小化提示词即可。
- **只做 OJ MVP**（假设题目与 07 已就绪）：可把上面提示词中的「第一部分」改为「若 07/08/09 与 problems/ 已存在则直接使用，否则只做最小必要文档与 1～2 道示例题」，其余只执行第二部分。
