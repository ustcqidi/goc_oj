# GOC 实施计划：先出题，后 Online Judge

你当前带儿子学 GOC 的两个痛点：

1. **题库太少**：只能找历年比赛题，希望有「出题专家」能持续生成新题，训练逻辑思维、编程思维，并与小学初中数学结合。
2. **没有 Online Judge 系统**：题库、做题记录、编译器、判题等一整套尚未实现。

本计划**分步实施**：**先解决问题 1（出题与题库扩充）**，再在题目和格式稳定的基础上解决问题 2（OJ 系统）。  
你后续可按照本计划的步骤，用 Claude / 代码逐步实现。

---

## 一、阶段划分

| 阶段 | 目标 | 本计划是否覆盖 |
|------|------|----------------|
| **阶段一** | 建立「出题能力」：题目标准格式、分类体系、出题规范、批量生成题目并落盘 | ✅ 下文详细拆解 |
| **阶段二** | 实现 OJ 系统（题库导入、做题记录、编译器、判题等） | ❌ 仅在本计划末尾列出依赖与衔接点 |

## 二、阶段一详细步骤（先做）

### Step 1：收集并分析题目案例，学习题目的大体模式

**目标**：在定义 Schema 和出题规范之前，先基于你提供的真实题目（如历年比赛题）提炼「题目长什么样」——题干结构、难度感觉、常用指令、与数学的对应等，避免闭门造车。

**你的动作**：在 `problems/samples/` 下为每道题建一个子文件夹（如 `001-螺旋三角形/`），放入 `description.md`（题目描述）和 `output.png`（期望输出图片）。数量不要求多，覆盖不同难度和类型即可（例如入门 2～3 道、进阶 2～3 道）。

**Claude 的动作（每收到新题自动执行）**：
  1. 读取 `description.md` 和 `output.png`，理解题目目标
  2. 补写参考 solution 到 `description.md`
  3. 将该题的出题模式（图形类型、数学对应、考察点、可扩展方向）追加到 `06-GOC 题目案例与模式提炼.md`

**出题产出物（Claude 批量出题时）**：
  - `description.md`：题目描述、详细绘图要求（颜色/线宽/转角/边长规律/循环约束等）、难度、标签、参考答案
  - `output.png`：根据 solution 用 Python 超采样渲染生成（4倍分辨率 + LANCZOS 缩小，消除锯齿）
  - 出题后立即保存到 `problems/samples/题号-题目名/` 目录，不得遗漏

**出题原则**：
  - 出题模式（06 文档）是知识框架，不是出题模板——禁止照抄模式直接套题
  - 题目形式要**创新多样**：同样考循环和角度，但视觉呈现、指令组合、数学背景可以完全不同
  - 类比高考命题：知识点固定，但题目情境、图形形态、考察角度要新颖

**产出物**：

- 文件：`06-GOC 题目案例与模式提炼.md`
- 内容：
  - **逐题分析**：每道样例题的图形解析、核心指令、数学对应、solution
  - **模式提炼**：归纳题干常用结构、常见题型分类、常考指令组合、与小学初中数学的对应点
  - **出题专家知识库**：供后续批量出题时作为上下文输入

**实现方式**：你放入图片和描述 → Claude 读图补全 solution + 持续更新模式提炼文档。

### Step 2：定义「单道题目」的标准格式（Schema）

**目标**：让每道题都有统一、可被程序读写的结构，方便后续批量生成和将来导入 OJ。

**依据**：Step 1 提炼的题目形态 + `03-GOC 判题教学系统.md` 中的题目结构（id, title, description, difficulty, starterCode, solution, testcase）。

**产出物**：

- 文件：`07-GOC 题目数据格式（Schema）.md`
- 内容建议：
  - 必填字段：`id`、`title`、`description`、`difficulty`（easy/medium/hard）、`starterCode`、`solution`
  - 可选字段：`tags`（如 几何、循环、小数数学）、`hint`、`expectedTrajectory`（可先留空，等有执行器再补）
  - 若用 JSON 存储：给出完整 JSON 示例 + 字段说明；若用 Markdown + frontmatter 也可一并约定
  - Schema 中的 description 结构、难度取值等，应与 Step 1 案例中的实际形态一致

**实现方式**：写一份 Markdown 文档即可；若有 JSON Schema（.json schema 文件）可一并给出，便于校验。

---

### Step 3：建立题目分类与难度体系

**目标**：为「出题」和「选题」提供维度：按知识点、按难度、按与数学/逻辑的对应关系。

**产出物**：

- 文件：`08-GOC 题目分类与难度体系.md`
- 内容建议：
  - **难度**：入门 / 进阶 / 高级（与 03 中 easy/medium/hard 对应）；具体描述尽量与 Step 1 案例中的难度感受一致
  - **知识点/标签**（与 01 指令、小学初中数学对应）：
    - 入门：直线与移动（fd/bk/rt/lt）、抬笔落笔、颜色与粗细
    - 进阶：循环（for）、重复图形、正多边形、简单规律
    - 高级：条件（if/else）、变量与计算、多图形组合、坐标（moveTo/lineTo）
  - **与数学的对应**：角度与正多边形、对称、周长与面积（描述性）、简单数列/规律等，便于按「数学主题」选题；可引用 Step 1 案例里出现的数学点

**实现方式**：写一份 Markdown 文档，列出分类表与标签表；后续出题时每题打上 `tags` 和 `difficulty`。

---

### Step 4：编写「出题专家」规范（出题指南）

**目标**：让人或 AI 按同一套规则出题，保证题目质量、难度递进、且与逻辑/数学结合。

**产出物**：

- 文件：`09-GOC 出题专家规范（出题指南）.md`
- 内容建议：
  - **题目形态**：题干（description）怎么写、是否需要输入输出（cin/cout）、是否限制只能用某类指令（如必须用 for）；**以 Step 1 案例的题干风格为范本**
  - **难度控制**：入门/进阶/高级各应覆盖哪些指令与思维点
  - **与小学初中数学的结合**：角度、正多边形内角、对称、简单数列、坐标与图形等，各对应哪些 GOC 题 type
  - **参考 01 指令集**：只使用 01 中已定义的指令，避免超纲
  - **标准答案与 starterCode**：solution 必须可运行且符合 01 规范；starterCode 可留空或给部分代码
  - **判题要点**（为后续 OJ 预留）：本题是「轨迹一致」即可，还是「必须含 for」等结构要求，可简短注明

**实现方式**：写一份 Markdown；可再抽成「出题 checklist」，供每次出题或 AI 生成时对照。

---

### Step 5：批量生成题目（含题干、starterCode、solution）

**目标**：在 Step 1～4 的基础上，生成一批符合规范的 GOC 题目，覆盖不同难度与标签；风格与难度参考 Step 1 案例。

**产出物**：

- 题目库目录：例如 `goc/problems/` 或 `goc/questions/`
- 每题一个文件，格式按 Step 2 的 Schema（07）：
  - 推荐：`problems/001-画正方形.json`（或 `001-画正方形.md` 含 frontmatter）
  - 内容包含：title, description, difficulty, tags, starterCode, solution；expectedTrajectory 可先省略，等有执行器再补

**实现方式**（可任选或组合）：

- **方式 A（人工 + AI）**：你或 Claude 按 08 出题指南逐题写，保存为 JSON/MD。
- **方式 B（半自动）**：写一个「题目模板 + 参数」的脚本（如用 YAML/JSON 描述多道题的差异），由脚本生成多份题目文件。
- **方式 C（AI 批量生成）**：把 01 指令规范 + Step 1 案例与模式提炼 + 07 Schema + 08 分类 + 09 出题指南 作为上下文，让 Claude 一次生成 N 道题（如每难度各若干道），你审阅后按 Schema 格式保存到 `problems/`。

**建议**：先做 10～20 道题（入门为主、少量进阶），验证格式与难度是否合适，再扩大规模。

---

### Step 6：题目索引与检索方式（可选但推荐）

**目标**：便于按难度、标签、数学主题筛选题目，也便于后续 OJ 导入。

**产出物**：

- 文件：`problems/index.json`（或 `problems/README.md`）
  - 索引内容：每题 id、title、difficulty、tags、文件路径
- 若有脚本：可根据 `problems/` 下文件自动生成 index，保证与真实题目一致

**实现方式**：手写一份 index 或写一个小脚本（如 Node/Python）扫描 `problems/*.json` 生成 index.json。

## 四、与阶段二（OJ 系统）的衔接

完成阶段一后，你会有：

- 从真实案例提炼的题目模式（06）
- 明确的题目 Schema（07）
- 分类与难度体系（08）
- 可持续复用的出题规范（09）
- 一批可直接使用的题目文件（04 的「题库数据模型」可引用 07）

阶段二实现 OJ 时将会用到：

- **题库**：直接读取 `problems/` 下文件或 `index.json` 导入。
- **判题**：题目中的 `solution` 可在有执行器后跑出 `expectedTrajectory`，或由判题服务在首次导入时生成并回写。
- **04 实现缺口清单**：OJ 相关的技术文档（技术架构、执行轨迹规范、解析与执行引擎、判题 API 与逻辑等）在阶段二再按 04 的列表逐步补齐。阶段一已占用编号 06～09（案例与模式、题目格式、分类、出题规范），阶段二新建技术文档时可从 10 起编（如 10-技术架构与选型、11-执行轨迹规范等），避免编号冲突。

## 五、建议实施顺序（给 Claude / 你）

1. **Step 1**：**你在 `problems/samples/` 下放入题目案例**（每题一个文件夹，含 `description.md` + `output.png`）→ Claude 读图补 solution、持续更新 `06-GOC 题目案例与模式提炼.md`，逐步建立出题专家知识库。
2. **Step 2**：依据 Step 1 的形态，写 `07-GOC 题目数据格式（Schema）.md`。  
3. **Step 3**：写 `08-GOC 题目分类与难度体系.md`，与案例中的难度和类型对齐。  
4. **Step 4**：写 `09-GOC 出题专家规范（出题指南）.md`，以 Step 1 案例为范本。  
5. **Step 5**：创建 `problems/`，按 07 的 Schema 生成第一批题目（建议 10～20 道，以入门为主）。  
6. **Step 6**：为 `problems/` 建立 `index.json` 或 README 索引。  

每步做完可在本计划里打勾或记一笔，再进入下一步。**第一步是关键**：你提供案例 → 学习题目模式 → 后续 Schema、分类、出题指南都贴着真实题目来，生成的新题才会像「比赛题风格」。需要批量出题时，把 01、06（案例与模式）、07、08、09 一起给 Claude，即可按「出题专家」角色生成新题。
