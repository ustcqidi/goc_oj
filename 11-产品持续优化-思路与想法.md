# GOC 产品持续优化：思路与想法记录

本文档用于**记录和整理**各种产品优化思路。  
**与《12-系统打造过程与AI-Agent实践记录.md》的关系**：本文档只记「优化项」；12 记整体打造过程与 AI Agent 实践。若某次讨论同时涉及新优化思路和过程/决策，会同步更新两处。你可以随时抛出想法，我会以**产品专家 + UE + 资深软件工程师**的视角和你一起讨论，并把共识或待办整理进本文档，便于后续排期与落地。

---

## 一、如何使用这份文档

- **你**：提出想法（功能、体验、性能、技术、内容等均可）。
- **我**：从三个角度回应并整理进文档：
  - **产品**：价值、优先级、与目标用户（孩子/家长/老师）的匹配度；
  - **UE**：交互、反馈、可发现性、学习曲线；
  - **工程**：实现成本、技术债、依赖、风险。
- **结果**：每条想法会归到下面某一类，并标上状态：`[待讨论]` / `[已采纳]` / `[已落地]` / `[暂缓]`，必要时附简短结论或下一步。

文档会随讨论**持续更新**，无需一次写全。

**题目收集与题库完善**的完整 workflow 见 **《13-题目收集与题库完善Workflow.md》**；**技术栈与部署**的适用性、升级时机见 **《14-技术方案评估.md》**。本文档只记产品优化思路。

**编号约定**：每条优化项分配一个 **GOC-XXX** 编号（如 GOC-001），便于沟通时引用（如「先做 GOC-028」）。

### 附：GOC-006 / GOC-029 / GOC-030 实施结论

以下为讨论后整理的实施顺序、要点与风险，供实现与验收对照。

**实施顺序建议**：**006 → 030 → 029**  
先保证图形「画得全、看得全」（006），再减轻拥挤感（030），最后统一视觉语言（029）。

| 顺序 | ID | 目标 | 实施要点 |
|------|-----|------|----------|
| 1 | **GOC-006** | 画布与目标图形居中、不截断 | ① 根据轨迹计算 bounding box；② 算 `scale = min((canvasW-留白)/(bboxW), (canvasH-留白)/(bboxH))` 及居中偏移；③ 主画布与「目标图形」canvas **共用同一套适应逻辑**，渲染前 `ctx.setTransform(scale,0,0,scale,tx,ty)` 或等价；④ 轨迹为空/单点时 fallback 为只居中不缩放或占位文案。 |
| 2 | **GOC-030** | 排版与留白，减轻拥挤 | ① 题目描述 / 目标图形 / 编辑区 / 画布 之间加大 margin（建议 16px～24px）或加分隔线；② **指令参考区默认收起**，仅保留「📖 指令」入口，点击后展开完整面板（分组保留：画笔/坐标/图形/控制）；③ 确保「📖 指令」入口显眼，避免用户找不到参考。 |
| 3 | **GOC-029** | 整体视觉与专业感 | ① 引入 CSS 变量（如 `oj/css/tokens.css` 或内联在 index 头部）：主色、辅色、中性灰、标题/正文/辅助字号、圆角；② 主按钮（运行、提交）统一主色实心，次按钮（重置、提示）outline 或浅底；③ 区块背景与 padding 统一（如 12px/16px）；④ 先只改 OJ 单页，不扩大范围。 |

**风险与注意**  
- **006**：确认执行器/渲染是否已有轨迹 bbox；若无则需在渲染前算一遍（可缓存，避免长轨迹重复计算）。  
- **029**：改动样式时避免破坏现有功能；可先小范围替换再逐步扩大。  
- **030**：指令区收起后，「📖 指令」需易于发现（如工具栏固定位置、文案清晰）。
