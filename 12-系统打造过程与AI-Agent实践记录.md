# GOC 系统打造过程与 AI Agent 实践记录

本文档记录 **GOC 出题 + Online Judge MVP** 是如何一步一步打造出来的，重点放在**思路、决策和 AI Agent 编程模式**上，便于与人分享「用 AI Agent 做产品」的完整过程。

---

## 使用说明

- **目标读者**：想了解「如何用 AI/Agent 从零搭出一个可用的教学产品」的人。
- **持续更新**：**后续所有对话中，我会提取核心思路并更新本文档**（新思路、新决策、新实践会追加到「后续对话提炼」或相应章节）。
- **与其它文档的关系**：00～11 是产品与规范本身；本文档是**过程与元叙事**，不替代具体规范。

---

## 一、起点：痛点与目标

- **场景**：带孩子学 GOC 图形化编程。
- **两个痛点**：
  1. **题库太少**：只能找历年比赛题，希望有可持续的「出题能力」，题目能训练逻辑/编程思维，并与小学初中数学结合。
  2. **没有 OJ**：缺题库、做题记录、编译器、判题等一整套系统。
- **产品目标**：先解决出题与题库，再在题目和格式稳定的基础上做 OJ，分阶段实施。

---

## 二、先搭「文档体系」再动手

- **已有基础**：01 指令规范、02 在线编程网站（产品设计）、03 判题教学系统（PRD），三者是「语言 → 界面 → 平台」的递进关系（见 00 文档说明）。
- **核心思路**：不急着写代码，先把「要做什么、做成什么样」写清楚，让 AI 和人都按同一套规范执行。
- **后续动作**：在 04 中列出实现缺口（技术架构、轨迹规范、判题 API 等），在 05 中写出「先出题、后 OJ」的详细步骤与交付清单。这样 Agent 或人都有明确的输入输出和验收标准。

---

## 三、出题：先学「题目长什么样」再定规范

- **思路**：避免闭门造车——先基于真实题目（你提供的 samples）提炼模式，再定义 Schema 和出题指南。
- **做法**：
  - 你在 `problems/samples/` 下放题目案例（description.md + output.png），覆盖不同难度与类型。
  - 由 AI 补写参考 solution、并把每道题的分析与「出题模式小结」写入 **06-GOC 题目案例与模式提炼.md**。
  - 再基于 06 的形态写 **07 题目数据格式（Schema）**、**08 分类与难度体系**、**09 出题专家规范**；出题时以 06 为范本，保证风格一致。
- **原则**：06 是「知识框架」不是「出题模板」——同一知识点要换情境、换图形、换数学角度，题目多样不雷同。

---

## 四、计划可执行化：步骤 + 交付清单 + 提示词

- **05 实施计划**：把阶段一拆成 Step 1～6（案例与模式 → Schema → 分类 → 出题指南 → 批量出题 → 索引），每步有目标、产出物、实现方式。
- **交付清单**：在 05 和「出题与 OJ 一次性完成」中明确 10 项交付物（06～10 文档、problems、index、执行器、前端、判题），便于验收。
- **提示词**：写出「总控任务」提示词（05-实施计划-Claude自主执行提示词.md），让单 Agent 在**不向人提问**的前提下，按顺序完成 Step 2～6，并做最终验收；同时提供单步提示词、断点续跑、验收专用提示词，方便分步或中断后继续。

---

## 五、一次性做完：出题 + OJ MVP

- **决策**：在「只做出题」之外，增加「出题 + OJ MVP 一次性完成」的选项，用一份长提示词让 Agent 在一个会话里完成阶段一 + 阶段二 MVP。
- **OJ MVP 范围**（见 05-实施计划-出题与OJ一次性完成.md）：
  - **纯前端、无后端、无登录**；题目与判题均在浏览器内完成。
  - 执行轨迹规范（10）→ GOC 执行器（JS）→ 用执行器跑每道题的 solution 得到「期望轨迹」（运行时生成，不要求 JSON 预填）→ 单页 OJ（题目列表、做题页、运行、提交、轨迹绘图、判题）→ 判题逻辑（轨迹比对 + 容差）。
- **实现要点**：判题不依赖 JSON 里的 expectedTrajectory，而是打开题目时执行 solution 并缓存轨迹，提交时再执行用户代码并比对，适合静态部署、无后端。

---

## 六、AI Agent 编程模式的几种用法

- **单 Agent + 长提示词**：一个 Claude 会话 + 一份「出题 + OJ 一次性执行」提示词，包办规划、规范、出题、执行器、前端、验收。适合快速交付、人手有限。
- **多 Agent / Teams**：拆成规划、需求/规范、出题专家、执行器、轨迹与判题、前端/OJ、验收等角色，按工单与依赖顺序执行；可用多会话或多 Agent 平台实现（见 05-Agent 架构设想）。
- **轻量折中**：保留「单会话」，但在提示词里显式列出步骤与交付清单，让 Agent 自检；或拆成 3 个团队（规范与出题 / 引擎与判题 / 前端与验收）分三轮跑。
- **本项目的实际路径**：采用单 Agent + 一次性提示词完成出题与 OJ MVP；Agent 架构文档用于「理解可拆成哪些角色」和后续扩展参考。

---

## 七、验收与持续优化

- **产品验收**：按交付清单逐项检查（06～10、problems、index、执行器、前端、判题），并写验收报告（见 验收报告-产品专家.md）；结论通过，并留下改进建议（题目与 OJ 数据同步、expectedTrajectory 表述、端到端自检步骤）。
- **持续优化**：建立 **11-产品持续优化-思路与想法.md**，按类型（体验、功能、编辑器、内容、技术、AI）记录想法，用「产品 + UE + 工程」三视角讨论，并标状态（待讨论/已采纳/已落地/暂缓），便于排期与迭代。

---

## 八、持续运营的完整 Workflow

在 MVP 交付后，**题目收集与题库完善**的端到端流程为：自动收集 sample → 理解 sample → 提炼模式更新文档(06) → 出题 → 更新题库。步骤说明、输入输出与依赖见 **《13-题目收集与题库完善Workflow.md》**；可落地为 Skill 链或分步提示词。

## 十、当前协作模式总结：可零干预自动化

本节总结我们做这个系统的**固定模式**：你几乎不用在「执行」环节干预，只需定方向、写/审文档、复制提示词。

### 10.1 模式是什么

| 环节 | 谁在做 | 产出 |
|------|--------|------|
| **想法** | 你（或和 AI 讨论） | 产品想法记入 **11**，标 GOC-XXX、状态（待讨论/已采纳/已落地/暂缓/不适用） |
| **共识** | 你 + AI（产品/UE/工程三视角） | 11 里该条标「已采纳」、必要时写讨论记录与分组（§六） |
| **任务规格** | 你或 AI | 单独一篇 **ClaudeCode 任务说明**（如 16、17、24、25～34）：硬性要求、输入输出、步骤、验收、**一句话任务** |
| **执行** | **Claude Code** | 读任务 doc + 07/01/10 等规范 → 按步骤做完 → 更新 index/sync/实现记录 → **一次性汇报**，不向你提问 |
| **验收** | 你（可选） | 看汇报或抽查；重要批次可写验收报告（如 docs/验收报告-17至22.md） |
| **状态同步** | 你或 AI | 把 11 里对应 GOC-XXX 改为「已落地」或「不适用（已回滚）」 |

你的干预集中在：**提想法、拍板采纳、看结果**；中间「怎么写代码、怎么改哪些文件、怎么跑脚本」全部交给 Claude Code 按文档执行。

### 10.2 为什么可以「不需要干预」

1. **文档即规格**：每个任务有独立文档（如 34），里面写清「硬性要求：全程自主、不要问我」、步骤 1～7、验收自检。Claude Code 只认文档，不依赖你口头发言。
2. **一句话任务**：每篇任务说明末尾有「复制给 Claude Code」的整段提示词，你**复制粘贴即下发**，无需再解释。
3. **规范统一**：07 Schema、01 指令、10 轨迹、15 硬性要求 在所有任务里一致引用，执行器与题目格式稳定，Agent 不会猜歧义。
4. **回滚也是任务**：不适用时不做「说服」，而是写**回滚任务说明**（如 24、33），同样用一句话任务交给 Claude Code 执行，保证可逆。

### 10.3 可进一步自动化的方向（可选）

若希望**连「写任务说明」都少动手**，可以在现有基础上加一层：

| 方向 | 做法 |
|------|------|
| **从 11 生成任务说明** | 写一个小流程或提示词：输入「11 中某条 GOC-XXX 已采纳」，输出一篇 34 风格的任务说明（含步骤、验收、一句话任务）；你只审阅后复制下发。 |
| **执行后自动更新 11 状态** | 在任务说明里约定：Claude Code 完成后除更新 实现记录 外，还输出「本次完成的 GOC-XXX 列表」；你或另一 Agent 据此把 11 里对应项改为「已落地」。 |
| **批量下发** | 合并多篇任务说明为一则提示词（如 32 合并 28～31），一次复制即让 Claude Code 按顺序执行多任务，你只收一份总汇报。 |
| **定时/触发跑任务** | 若有可调度的 Agent 环境，可把「按 34 出题」「按 16 跑 workflow」等设为定时或「新 sample 到位即触发」，完全无人值守（需注意仓库权限与冲突）。 |

当前已经做到：**你只负责「想做什么」和「复制哪段话」；「怎么做完」全由 Claude Code 按文档自主完成，无需你中途参与。**

### 10.4 当前产品迭代闭环（谁在做哪一步）

| 步骤 | 谁在做 | 说明 |
|------|--------|------|
| 1. 提出优化方向 | **你 或 Cursor 里的我（PM 角色）** | 想法记入 11，标 GOC-XXX |
| 2. 研究 UE / 技术方案 | **我（Cursor）** | 产品/UE/工程三视角讨论，生成或更新设计/方案文档 |
| 3. 生成 Claude Code 执行说明 | **我（Cursor）** | 写 16、17、24、25～34 类文档，含步骤、验收、一句话任务 |
| 4. 唤起 Claude Code 写代码 | **你** | 复制「一句话任务」到 Claude Code，由 Claude Code 执行 |
| 5. 验收 Claude Code 结果 | **你** | 看汇报、抽查、必要时写验收报告；把 11 里对应项标「已落地」 |
| 6. 下一轮优化 | **你 或 我** | 回到步骤 1 |

**结论**：我（Cursor 里的对话 Agent）**不能**直接调度或唤起 Claude Code。步骤 4 必须由你手动「复制提示词 → 在 Claude Code 里粘贴并运行」；步骤 5 的验收也需你（或你指定的流程）完成。因此当前是**半自动**：从「想法」到「可执行说明」我可以包办；「执行」与「验收」仍依赖你。

### 10.5 能否全自动化完成产品迭代？

| 目标 | 可行吗 | 条件 / 缺口 |
|------|--------|-------------|
| **全自动：从想法到代码到验收无人值守** | **目前不行** | 需要有一个**可被程序调用的执行层**：能根据任务说明自动唤起「写代码」的 Agent（Claude Code 或等价物），并在完成后把结果交给「验收」步骤（人或自动检查）。若你方有 Agent 编排/调度平台（如通过 API 或 MCP 触发 Claude Code），则可把步骤 4 自动化；验收仍可简化为「自动跑测试 / 检查清单」，关键决策是否自动合并到主分支。 |
| **半自动：你只做「提方向 + 粘贴一次 + 验收」** | **已实现** | 步骤 1～3 由我完成，你只负责在步骤 4 粘贴提示词、步骤 5 验收并更新 11，无需参与中间实现细节。 |
| **进一步：自动生成任务说明** | **可行** | 你只说「做 GOC-031」或「按 11 里已采纳的下一项」，我生成对应的 Claude Code 执行说明与一句话任务，你审阅后复制给 Claude Code，减少你写文档的负担。 |

若未来有 **MCP / API 可调用「代码执行 Agent」**，我可以把「一句话任务」通过工具发出去，并由同一轮对话根据返回结果更新 实现记录 或 11 状态，那时才有机会接近「全自动产品迭代」；目前在我的能力范围内无法代替你「唤起 Claude Code」这一步。

### 10.6 若全部由 Cursor 内的我写代码（不用 Claude Code）

| 问题 | 结论 |
|------|------|
| **能完全实现自动化吗？** | **可以。** 你只需说一句「做 GOC-XXX」或「按 11 里已采纳的下一项做」，我可以在同一轮对话内：读 11 与对应任务说明（如 25～34）→ 研究 UE/技术方案（必要时）→ **直接改代码、改 JSON、跑 sync/脚本** → 更新 实现记录 → 汇报。无需你再复制提示词到 Claude Code，也无需切换工具。 |
| **还缺什么才算「全自动」？** | 仅剩两处需人或机制参与：（1）**发起本轮**：由你说「做下一项」或由定时/事件触发（若 Cursor 支持）；（2）**验收/合并**：你确认结果、合并分支，或由 CI 通过后自动合并。若这两步也由规则或触发器完成，即趋近「零人工」产品迭代。 |
| **和「用 Claude Code」的对比** | 用 Claude Code：任务说明 + 一句话任务写好后，需你复制到 Claude Code 执行；验收在彼端。**用我写代码**：任务说明可保留为「规格」（我按 25～34 等文档执行），你只在我这里说一句，我从规格到代码到记录一条龙做完，你只做发起与验收。 |
