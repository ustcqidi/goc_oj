# GOC 项目技术方案评估

本文档评估当前 MVP 的技术栈是否适合后续功能扩展与部署上线，并给出阶段性建议。MVP 已完成；后续会持续加 feature 并部署，需提前看清边界与升级时机。

---

## 一、当前技术栈概览

| 层级 | 当前方案 | 说明 |
|------|----------|------|
| **前端** | 单页 HTML + 内联 CSS + 内联 JS | `oj/index.html` 一个文件承载布局、样式、逻辑；无 React/Vue 等框架 |
| **题目数据** | 内嵌 `oj/data/problems.js`（`window.PROBLEMS_DATA`） | 静态数据，与 `problems/*.json` 需手动或脚本同步 |
| **执行与判题** | 纯前端 `oj/js/goc-executor.js` | 词法/语法解析、执行、轨迹输出、Canvas 渲染、判题均在浏览器内完成 |
| **构建** | 无 | 无 npm/Vite/Webpack，无打包、无压缩 |
| **后端** | 无 | 无 API、无数据库、无用户体系 |
| **部署** | 静态文件 | 可 `file://` 打开或任意静态托管（如 Nginx、GitHub Pages、对象存储 + CDN） |

---

## 二、对 MVP 的适用性（结论：合适）

- **交付快**：无构建、无后端，改完即用，适合验证产品与教学场景。
- **零运维**：不依赖服务器、数据库，部署即丢到静态托管即可。
- **离线可用**：可完全本地打开，适合内网或无外网环境。
- **与目标一致**：当前功能（题目列表、做题、运行、提交、判题、localStorage）均可在该栈下实现。

**结论**：当前技术栈对 MVP 是合适的，无需为「只是 MVP」而立刻升级。

---

## 三、后续 Feature 与部署对技术栈的影响

以下按「加什么功能 / 做部署」看现有方案是否够用、何时需要升级。

### 3.1 已采纳或高优先级优化（11 文档内）

| 需求 | 当前栈能否支撑 | 说明 |
|------|----------------|------|
| 运行时分步绘制 + 倍速 | ✅ 能 | 纯前端：用 timer 按轨迹条播放 + 重绘 Canvas，无后端依赖 |
| 题目配图（output.png）+ OJ 展示示例图 | ✅ 能 | 题目数据增加图片路径或 URL，前端 `<img>` 展示；图存静态目录或 CDN |
| 指令参考可折叠列表、点击插入 | ✅ 能 | 前端增加一块 UI + 静态指令列表（JSON 或写死），插入到 textarea |
| 题目数据与 OJ 同步（生成 problems.js） | ✅ 能 | 用 Node 或脚本从 `problems/*.json` 生成 `problems.js`，无需改架构 |
| 语法高亮 / 简单自动补全 | ⚠️ 可做但会变重 | 用正则轻量高亮可保留无构建；若用 Monaco/CodeMirror 建议引入构建（npm） |

以上在**保持无后端、无构建**的前提下大多可完成；仅「上 Monaco/CodeMirror」时建议顺带引入构建与模块化。

### 3.2 部署上线（静态）

| 需求 | 当前栈能否支撑 | 说明 |
|------|----------------|------|
| 公网访问、HTTPS、域名 | ✅ 能 | 静态资源放任意静态托管 + CDN，绑定域名、上 HTTPS 即可 |
| 多环境（开发 / 预发 / 生产） | ✅ 能 | 不同目录或不同分支部署到不同地址即可；无环境变量时可用不同 `problems.js` 或配置 |
| 访问速度与缓存 | ✅ 能 | 静态资源天然适合 CDN 缓存；可后续给 HTML 设短缓存、JS/CSS/数据设长缓存 |

**结论**：仅做「静态站点部署上线」，当前技术栈足够，无需为部署单独加后端或构建。

### 3.3 可能的中长期 Feature（03 / 02 / 11 中提及）

| 需求 | 当前栈是否够用 | 建议 |
|------|----------------|------|
| 用户登录、提交记录上库、多设备同步 | ❌ 不够 | 需后端（API + 数据库）+ 鉴权；前端需区分「已登录/未登录」与请求封装 |
| 题目列表/详情由 API 返回（动态题库） | ❌ 不够 | 需后端或 BFF 提供题目列表、单题详情；前端改为 fetch 而非依赖内嵌 `problems.js` |
| 分享代码链接（只读） | ✅ 可做 | 方案 A：前端 hash/base64 带代码，无后端；方案 B：后端存 snippet 生成短链 |
| 关卡/学习路径、成就 | ⚠️ 可半做 | 仅前端 + localStorage 可做本地关卡与成就；若需跨设备、统计需后端 |
| AI 辅助（生成代码、解释） | ⚠️ 看形态 | 若调外部 API，前端即可；若自建模型服务，需后端或单独服务 |
| Step Run（手动单步、断点） | ✅ 能 | 执行器支持「执行到第 N 条暂停」+ 前端按钮即可，可仍为纯前端 |
| 多页路由（如独立「题目列表」「用户中心」） | ⚠️ 可选 | 单页内多视图即可；若希望独立 URL、分享到某页，可引入轻量路由或 SPA 框架 |

## 四、技术栈升级时机建议

- **继续用当前栈（无构建、无后端）**：适合只做 11 里已采纳的优化 + 静态部署上线，不做登录、不做动态题库、不做跨设备进度。可一直用到「需要用户体系或服务端判题」为止。
- **引入前端构建（如 Vite + npm）**：建议在以下任一成立时考虑：  
  - 要接 Monaco/CodeMirror 做语法高亮、补全；  
  - 希望拆分为多文件（组件化）、用 TypeScript、做资源压缩与打包；  
  - 需要环境变量或多环境构建（如不同 API 地址）。  
  引入后仍可保持「静态站点 + 无后端」。
- **引入后端（如 Node / Python + 数据库）**：建议在需要**用户登录、提交记录持久化、动态题库、或服务端判题/安全**时再上。可先定 API 契约（题目列表、提交、判题等），再选技术栈（Node + 轻量 DB 或 Python FastAPI 等）。
- **部署上线本身**：不依赖技术栈升级，用现有静态文件即可上线；后续若用 Vite 等，则部署前多一步 `build`，把产物部署到同一静态托管即可。

## 五、风险与约束（当前栈）

| 风险/约束 | 说明 | 缓解 |
|-----------|------|------|
| 题目数据双源 | `problems/*.json` 与 `oj/data/problems.js` 需同步 | 用脚本从 JSON 生成 problems.js，或改为前端 fetch 静态 JSON（需解决 file:// 或同源） |
| 单文件过大 | `index.html` 内联过多，后续难维护 | 适时拆出 CSS/JS 为独立文件；或引入构建与组件化 |
| 无类型与测试 | 纯 JS，无类型检查、无单测 | 后续若上构建可加 TypeScript、Vitest 等；执行器可单独抽为模块便于测 |
| 浏览器兼容 | 依赖现代浏览器 API（如 Canvas、ES6+） | 文档注明推荐 Chrome/Edge/Safari 较新版本；必要时加 polyfill |

## 六、总结表（何时用什么）

| 阶段 | 建议技术形态 | 部署方式 |
|------|----------------|----------|
| **当前（MVP + 已采纳优化）** | 保持静态 HTML + JS + 内嵌/静态数据；执行器、判题、绘图动画、指令参考、题目图均在前端完成 | 静态托管 + CDN，绑定域名 HTTPS |
| **需要更好编辑器体验时** | 引入 Vite + npm，可选 React/Vue；接入 Monaco 或 CodeMirror；题目数据可仍为静态 JSON 或后续接 API | 先 `build` 再部署静态产物 |
| **需要用户与持久化时** | 增加后端（API + DB），前端改为调用 API；鉴权、提交记录、题库管理走服务端 | 前端静态部署 + 后端单独部署（或 BFF 同域） |

**直接回答「当前技术栈合适吗」**：  
- **对 MVP 和即将做的优化、以及「只做静态部署上线」**：✅ 合适，无需立刻换栈。  
- **对「后续还有很多 feature、会部署上线」**：✅ 上线本身不要求换栈；若后续要做登录、动态题库、服务端判题等，再按上表引入后端与（按需）前端构建即可。建议把本文档当作「技术方案与升级时机」的单一入口，随 feature 决策持续更新。

## 七、到时需要推倒当前版本重写吗？

**不需要。** 当前版本可以在现有基础上**渐进演进**，而不是推倒重写。

### 7.1 加「前端构建」时

- **现有代码可保留**：`index.html` 可逐步拆成「入口 HTML + 若干 JS 模块」；`goc-executor.js` 已是 IIFE，可直接作为入口或改为 `export` 被引用。
- **做法**：用 Vite（或类似）把当前 `oj/` 目录当作源码，先只做「打包、压缩、可选的 TS」，页面结构和逻辑不变；再按需引入 Monaco、Vue/React 等，是**增量替换**（例如先只换编辑器区域），不是整站重写。

### 7.2 加「后端」时

- **后端是新增一层**：API + 数据库是全新服务，不替代现有前端，只是前端多了一种「数据从哪来、提交往哪发」的方式。
- **前端改动是「换数据源与请求」**：
  - 题目列表：从「读 `window.PROBLEMS_DATA`」改为「`fetch('/api/problems')`」，渲染逻辑可不变。
  - 提交：从「仅本地判题 + localStorage」改为「请求 `POST /api/submit`，后端存记录、可选服务端判题」，判题结果仍可沿用现有展示方式。
  - 登录：增加「登录态、请求头带 token、未登录时跳转或提示」，不要求重写整页。
- **执行器与判题**：可继续在前端跑（现有逻辑保留）；若未来要做服务端判题，后端可复用同一套执行逻辑（如 Node 版 goc-executor），前端仍可保留「即时运行+绘图」不变。

### 7.3 小结

| 升级项 | 是否推倒重写 | 实际做法 |
|--------|----------------|----------|
| 引入 Vite/打包 | ❌ 否 | 以现有 `oj/` 为源码渐进加构建，拆模块、换编辑器等可逐步做 |
| 引入后端 API | ❌ 否 | 后端新建；前端改为调用 API 获取题目、提交记录，并增加鉴权，逻辑与 UI 可大部分复用 |
| 换框架（如 React） | ⚠️ 可选 | 可只在新页面或新模块用；现有页面保留，或按需逐块迁移，不必一次性重写 |

结论：**当前版本是可持续演进的基础，不是一次性原型。** 后续加构建、加后端，都是「在现有版本上叠一层、改数据源与请求」，而不是推倒重写。
