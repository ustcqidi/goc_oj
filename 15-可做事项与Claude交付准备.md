# GOC 可做事项与 Claude Code 交付准备

本文档从产品文档中提炼**可直接交给 Claude Code 实现的任务**，并写明每项**交付前需要准备的内容**，便于你复制任务描述与必读文档给 Claude。

---

## 硬性要求：全程自主、不停不问

交给 Claude Code 的**任意任务**（单任务如 A1～A8，或全量 Workflow A9），均须满足：

- **全部由 Claude Code 自己做完**：选定任务所包含的所有步骤在一次会话内完成，不要中途停下来等用户。
- **不要用户干预**：不向用户提问、不要求用户确认、不等待用户输入；遇到歧义或可选情况时，按本文档及对应规范（如 16、07、02）的约定自行决定。
- **不要跳过步骤**：按任务说明或 16 的五步顺序执行到底，做完后再按 §三 留下交付物与留存文档，最后一次性向用户汇报结果。

下发任务时请把本段要求一并带给 Claude Code（或说明「按 15 文档中的硬性要求执行」）。

## 二、交给 Claude Code 前需要准备的内容

### 2.1 通用准备（所有任务都建议带上）

1. **仓库与路径**
   - 说明：本项目在 `goc/` 下，OJ 在 `goc/oj/`，题目在 `goc/problems/`，文档在 `goc/*.md`。
   - 可写一句：「代码与数据都在仓库的 `goc/` 目录下，请在该目录内读写文件。」

2. **必读文档（按任务选读）**
   - **01-GOC 指令规范.md**：语言与 pen 方法，任何改执行器/轨迹/题目的任务都要。
   - **07-GOC 题目数据格式（Schema）.md**：题目 JSON 字段、命名规则。
   - **10-执行轨迹规范.md**：轨迹格式，判题、同步、expectedTrajectory 相关必读。
   - **02-GOC 在线编程网站.md**：前端布局与功能描述，做 OJ 前端改动用。
   - **11-产品持续优化-思路与想法.md**：每项优化的产品/UE/工程说明与备注。
   - **14-技术方案评估.md**：当前无构建、无后端，纯静态；做前端时不要引入 npm/React 除非你明确要求。

3. **现有实现**
   - 执行器：`oj/js/goc-executor.js`（解析、执行、轨迹、渲染、判题）。
   - 题目数据：`oj/data/problems.js`（`window.PROBLEMS_DATA`）；`problems/*.json` + `problems/index.json`。
   - 无构建、无后端：保持单页 HTML + 内联/独立 JS，可新增 JS 文件。

## 三、Claude Code 交付物与留存文档

完成任意任务后，除代码与数据外，**必须留下可追溯的文档**，便于后续维护与交接。交付时请同时满足以下约定。

### 3.1 交付物总则

| 类型 | 要求 |
|------|------|
| **代码 / 脚本** | 按任务产出（见 §1.1 表）；所有新增或修改的文件放在 `goc/` 下约定路径，不散落无说明文件。 |
| **留存文档** | 见下 3.2，**至少完成其中一项**；若本次同时改多处（如一批 A2+A6+A7），建议写一份「本次实现说明」覆盖全部改动。 |

### 3.2 必须或建议留存的文档

以下文档需落在 `goc/` 目录内（或 `goc/docs/`、脚本同目录），由 Claude Code 在交付时创建或更新。

| 场景 | 文档 | 内容要求 |
|------|------|----------|
| **新增脚本**（如 A1 同步脚本、A8/A9 抓取或 Workflow 脚本） | 同目录下 `README.md` 或脚本所在目录的 `README.md` | 脚本用途、入参/环境（如 Node 版本）、如何运行（命令示例）、输入输出路径；若依赖第三方包，列出并说明安装方式。 |
| **单任务交付**（如只做了 A2 或只做了 A4） | `goc/docs/实现记录.md`（若不存在则创建；可按日期追加）或 `goc/oj/README.md`（若仅改 OJ） | 本次完成的任务编号（如 A2）、修改了哪些文件、关键逻辑一句话；如何验收（如「打开 OJ 运行代码看动画」）。 |

### 3.3 文档放置约定

- 若已有 `goc/docs/`：上述「实现记录」「Workflow 实现说明」一律放在 `goc/docs/` 下。
- 若尚无 `goc/docs/`：由 Claude Code 在交付时创建 `goc/docs/` 并写入对应文档。
- 脚本自说明：若脚本在 `goc/scripts/`，则 `goc/scripts/README.md` 描述本目录下所有脚本；若脚本在项目根或 `goc/` 根，则在**该脚本同目录**放简短 README 或至少在脚本文件头部用注释说明用途与用法。

以上约定在给 Claude Code 的任务说明中一并给出（如复制 §三 或本段），以便交付时自动带上文档。

## 六、与 Claude Code 的共识与常见问题

以下为 Claude Code 实现时提出的问题及**已采纳的结论**，写进本文档后实现按此执行即可，无需再问。

### 6.1 配图（output.png）的生成方案

- **方案 A**：OJ 里增加「示例图」区域，打开题目时在**第二个 Canvas** 上跑一遍 solution 即时渲染——无需任何图片文件，当前架构直接支持。
- **方案 B**：用 Puppeteer/Playwright headless 跑截图，写回 `problems/<id>/output.png`——需额外脚本和 Node 环境，得到真正的静态图片文件。

**结论**：**先做方案 A**（成本低、立刻可用）；方案 B 留后续（如需要静态图做分享、CDN 或离线时再做）。

### 6.2 优先实现哪几个「已采纳」项

采纳 Claude Code 建议顺序，见上文 **§五、建议实现顺序**：第 1 批 = 运行动画+倍速 + OJ 示例图（方案 A）+ 指令参考面板；第 2 批 = 错误行号高亮；第 3 批 = 题目数据同步脚本。你可按需微调批次或顺序。

### 6.3 自动收集 sample 的「Skill」指什么？现在做还是等 OJ 优化完？

- **形式**：  
  - 若在 **Claude Code** 里用：可以是 `.claude/commands/` 下的 command（一个 md 文件），里层调用「抓取 + 写 samples + 更新清单」的**独立脚本**（Node 或 Python）。  
  - 若在 **Cursor** 里用：是 `.cursor/skills/` 下的 Skill。  
  - 文档 16 里的五步 Workflow 是按**脚本顺序执行**设计的，不依赖具体 IDE；「自动收集」这一步本质就是：读清单 → 请求 URL → 解析 HTML → 写 `samples/` + 更新清单。实现成**独立可执行脚本**即可，需要时再在 Claude Code 里用 command 调用该脚本。

- **时机**：  
  自动收集（扩充题库数据）与 OJ 前端优化（运行动画、示例图、指令参考等）**互不阻塞**，可并行。建议：**先做 OJ 第 1 批优化**（体验立竿见影），再跑 16 的 Workflow（含自动收集）扩充题目；若你更缺题，也可以先跑一轮 Workflow 再优化 OJ。
