# Claude Code 全量任务说明：题目收集与题库完善 Workflow

**使用方式**：将本文档（及下方「必读文档」中的规范）一起交给 Claude Code，要求其**按顺序完成全部任务，无需用户干预**。所有路径、格式、编号规则、异常处理均在本文档中约定，实现时不得向用户提问，按约定执行即可。

---

## 硬性要求：全程自主、不停不问

- **全部由 Claude Code 自己做完**：从步骤 1 到步骤 5，所有事情在单次会话内完成，不要中途停下来等用户。
- **不要用户干预**：不向用户提问、不要求用户确认、不等待用户输入；遇到歧义或可选情况时，按本文档约定自行决定（如抓取失败则跳过并写 workflow.log、清单只处理前 5 条等）。
- **不要跳过步骤**：五步顺序执行，一步完成再进入下一步，直到步骤 5 写完 `index.json` 和 `oj/data/problems.js` 并自检验收清单后，再向用户汇报结果。

## 一、任务总览与执行顺序

请按以下顺序**一次性完成**，中间不暂停、不向用户确认：

| 顺序 | 任务 | 输入 | 产出 |
|------|------|------|------|
| 1 | 自动收集 sample（抓取题目来源清单中的「待抓取」题目） | `problems/题目来源清单-noi.hnai.net.md` | `problems/samples/编号-题目名/`（description.md + 可选 output.png）；清单中成功抓取的行状态改为「已抓取」 |
| 2 | 理解 sample（分析新抓取的题目） | 上一步新增的 sample 的 description.md | 为每道新题生成「单题分析」内容（格式见下） |
| 3 | 提炼模式并更新 06 | 当前 samples 集合 + 步骤 2 的分析 | 更新 `06-GOC 题目案例与模式提炼.md`（追加新题章节 + 更新出题模式小结） |
| 4 | 出题（根据 06 模式生成新题） | 06 + 07 + 08 + 09 + 01 | 新增 1～2 道题目 JSON（如 `problems/016-xxx.json`） |
| 5 | 更新题库与 OJ | 所有 `problems/*.json` | 更新 `problems/index.json`；生成 `oj/data/problems.js` |

**本次运行范围约定**：

- **步骤 1**：只处理题目来源清单中**前 5 条**状态为「待抓取」的题目；若不足 5 条则全部处理。某条 URL 抓取失败（网络错误或无法解析）时跳过该条、不更新其状态为已抓取，并可将失败信息写入 `goc/workflow.log`（若文件不存在则创建），然后继续下一条。
- **步骤 2～3**：仅针对**本次步骤 1 新写入**的 sample 目录进行分析与 06 更新。
- **步骤 4**：根据当前 06 内容生成 **1 道**新题（若 06 中已有足够模式可再生成 1 道，共最多 2 道）。
- **步骤 5**：对所有 `problems/*.json` 重新生成 index 与 problems.js（包含已有题 + 步骤 4 新增题）。

## 四、步骤 1：自动收集 sample

### 4.1 输入

- 文件：`goc/problems/题目来源清单-noi.hnai.net.md`
- 表格格式：每行 `| 题目名称 | URL | 状态 |`；只处理状态为「待抓取」的行，且**仅处理前 5 条**（不足 5 条则全部）。

### 4.2 动作

1. 依次请求每条「待抓取」的 URL（GET），获取 HTML。
2. 从页面中抽取：**题目标题**、**题目描述/题干**、**参考代码**（如 `<pre><code>` 或含 GOC 的代码块）、**结果图**（若有 `<img>` 且明显为运行结果或目标图形，可下载为该题的示例图）。
3. 为每条成功抓取的题目：
   - 确定 sample 编号：当前 `problems/samples/` 下已存在目录名为 `XXX-*` 的，取最大编号 `N`，新题为 `N+1`，不足 3 位则前补 0（如 004、005）。若目录为空或仅有非数字前缀，则从 004 起编（已有 001、002、003 时）。
   - 题目名：由题目标题简化而来（去掉「GOC题解：」等前缀，保留 4～8 字中文），作为目录名后半部分，如 `004-放飞气球`。
   - 新建目录：`goc/problems/samples/编号-题目名/`。
   - 写入 `description.md`，格式见 4.3。
   - 若有结果图，保存为 `output.png`（或与页面一致的后缀，若为其他格式可统一为 png 或保留原格式）。
   - 将该行在清单中的状态改为「已抓取」并写回 `题目来源清单-noi.hnai.net.md`。
4. 若某 URL 请求失败或无法解析出标题/题干/代码，则**不**创建目录、**不**改状态，可选写入 `goc/workflow.log` 记录该 URL 及原因，然后处理下一条。

### 4.3 description.md 格式

与现有 `problems/samples/001-螺旋三角形/description.md` 一致，必须包含以下小节（无内容可写「无」或「na」）：

```markdown
# 编号 - 题目标题

## 题目描述
（题干文字）

## 要求
（若有则写，否则 na）

## 期望输出
见 `output.png`（或有图时类似说明；无图可写「无」）

## 难度
入门 / 进阶 / 高级（根据题目内容判断）

## 标签
`标签1` `标签2`（参考 08，空格分隔）

## 参考答案
```cpp
（GOC 代码，符合 01 规范）
```
```

## 五、步骤 2：理解 sample

### 5.1 输入

仅针对**步骤 1 在本轮中新建**的 sample 目录（即本次新抓取的几道题），读取其 `description.md`（若有 `output.png` 可参考，无则仅依据文本）。

### 5.2 动作与产出

为每道新 sample 生成一份「单题分析」，结构严格按 `06-GOC 题目案例与模式提炼.md` 中「001 - 螺旋三角形」的写法，包含：

- **图形分析**：一段话描述图形外观与构成规律。
- **核心指令**：用到的 pen 方法与控制流（如 `for`、`if`）。
- **考察点**：列表，如循环变量、转角、等差数列等。
- **数学对应**：表格，数学概念 ↔ 题目中的对应关系。
- **参考 Solution**：与 description.md 中参考答案一致的代码块（若 description 中无或不全，可根据题目要求补写，且必须符合 01）。
- **可扩展出题方向**：表格，变体 / 改动点 / 难度变化。

产出为**内存中的结构化内容**或临时文本，供步骤 3 直接使用，无需单独落盘（步骤 3 会写入 06）。

## 六、步骤 3：提炼模式并更新 06

### 6.1 输入

- 当前 `goc/06-GOC 题目案例与模式提炼.md` 的完整内容。
- 步骤 2 为**本轮新 sample** 生成的所有「单题分析」。

### 6.2 动作

1. 在 06 文档中，在最后一个「## 编号 - 题名」之后、「出题模式小结」之前，**按编号顺序**追加本次新题的章节，每节结构同 06 现有「001 - 螺旋三角形」：
   - `## 004 - 题名`（与 sample 目录名一致）
   - 图形分析、核心指令、考察点、数学对应、参考 Solution、可扩展出题方向。
2. 更新文档末尾的「出题模式小结」：
   - **已发现题型**：若本次新题属于新题型，在表中增加一行（题型 / 特征 / 代表题 / 核心指令）。
   - **常见题干结构**：若有新的题干句式可补充。
   - **难度标志**：若需可微调，保持与 08 一致。
3. 将修改后的完整内容写回 `goc/06-GOC 题目案例与模式提炼.md`。

## 七、步骤 4：出题

### 7.1 输入

- `goc/06-GOC 题目案例与模式提炼.md`（含出题模式小结与可扩展方向）。
- `goc/07-GOC 题目数据格式（Schema）.md`、`goc/08-GOC 题目分类与难度体系.md`、`goc/09-GOC 出题专家规范（出题指南）.md`、`goc/01-GOC 指令规范.md`。

### 7.2 动作

1. 根据 06 中的题型与「可扩展出题方向」生成 **1 道**新题（若 06 内容足够且不重复，可再生成 1 道，最多共 2 道）。要求：
   - 题干**不照抄** 06 中任何一道样例题；换情境、换图形或换数学角度。
   - 符合 09：任务说明、绘图要求清单、禁止直接给出解题代码。
   - 难度与标签符合 08；solution 仅使用 01 中已定义的指令。
2. 题目编号：读取 `goc/problems/index.json`，取当前最大 `id`（数字），新题 id 为 `最大 id + 1`，不足 3 位前补 0（如 016、017）。文件名：`编号-题目名.json`，题目名 4～8 字中文。
3. 每道题 JSON 必填字段：`id`、`title`、`description`、`difficulty`、`tags`、`starterCode`、`solution`；可选：`hint`、`constraints`、`expectedTrajectory`（可先写 `[]`）。
4. 将新题 JSON 写入 `goc/problems/编号-题目名.json`。

## 八、步骤 5：更新题库与 OJ

### 8.1 输入

- `goc/problems/` 下所有 `*-*.json` 题目文件（按文件名排序或按 id 排序）。
- 现有 `goc/problems/index.json` 结构（见下）。

### 8.2 动作

1. **更新 index.json**：  
   遍历所有题目 JSON 文件，生成索引数组，每项包含：`id`、`title`、`difficulty`、`tags`、`file`（文件名，如 `016-新题名.json`）。写回 `goc/problems/index.json`，保持 JSON 数组格式。
2. **生成 oj/data/problems.js**：  
   读取所有题目 JSON，组装为与当前 OJ 一致的格式：`window.PROBLEMS_DATA = [ ... ];`，数组中每项为题目完整 JSON（含 id、title、description、difficulty、tags、starterCode、solution、hint、constraints、expectedTrajectory 等）。写回 `goc/oj/data/problems.js`，确保 OJ 页面能直接使用该变量。

### 8.3 验收

- 打开 `goc/oj/index.html`（或通过静态服务访问），题目列表应包含原有题与步骤 4 新增题，且无重复、无缺字段。
- `problems/index.json` 与 `problems/*.json` 一一对应。

## 九、技术约束与实现方式

- **无需用户输入**：所有数量、路径、格式均按本文档执行；不要在中途询问用户。
- **可选用技术**：步骤 1 抓取可用 Node（如 node-fetch/axios + cheerio/jsdom）或 Python（requests + BeautifulSoup）等；步骤 2～4 为文本/JSON 生成与文件读写；步骤 5 为脚本读取 JSON 并写 index 与 problems.js。推荐在同一仓库内用单一脚本或少量脚本（如 `goc/scripts/workflow.js` 或 `workflow.py`）顺序执行 1～5，便于日后「一键运行」。
- **清单写回**：步骤 1 中每次将某行状态改为「已抓取」后，即可写回清单文件（或在本轮 5 条全部处理完后统一写回），确保 Markdown 表格格式不被破坏。
- **编码与换行**：所有写入文件使用 UTF-8；description.md、06 等为文本，保持合理换行。
