# Claude Code 任务说明：按出题建议（大纲学段 + 真题改编）出题

> 根据 `docs/出题建议-按大纲学段.md` 与 `docs/出题建议-真题改编.md` 生成题目 JSON 并入库。全程自主执行，不需用户干预。

---

## 一、硬性要求

- **全部由 Claude Code 自己做完**：本任务所包含的所有步骤在一次会话内完成，不要中途停下来等用户。
- **不要用户干预**：不向用户提问、不要求用户确认；遇到歧义时按本文档及 07 Schema、两份出题建议文档的约定自行决定。
- **不要跳过步骤**：按「任务概述 → 执行步骤」执行到底，完成后更新 `goc/docs/实现记录.md` 并汇报题目数量与分布。

---

## 二、任务概述

| 项 | 说明 |
|----|------|
| **输入** | `goc/docs/出题建议-按大纲学段.md`（P-001～P-060，60 题）、`goc/docs/出题建议-真题改编.md`（T-01～T-48，48 题） |
| **输出** | 新增题目 JSON 文件（`problems/编号-题目名.json`）、更新 `problems/index.json`、运行 sync 生成 `oj/data/problems.js` |
| **工作目录** | 仓库下的 `goc/` |
| **编号规则** | 当前题库最大编号为 **150**。新题从 **151** 起连续编号。 |

**合并策略：**

1. **第一批**：按「出题建议-按大纲学段.md」中的 **P-001～P-060** 依次生成 60 道题，编号 **151～210**（P-001→151，P-002→152，…，P-060→210）。
2. **第二批**：按「出题建议-真题改编.md」中的 **T-01～T-48** 逐题检查：若与已存在题目（001～150 及 151～210）**题意与输入/输出实质相同**（如均为「输入两数输出和」），则**跳过**；否则生成新题，编号从 **211** 起连续（211、212、…）。

## 四、题目 JSON 要求（07 Schema）

每道题一个文件：`problems/编号-题目名.json`，例如 `151-加法计算器.json`。

### 4.1 必填字段

| 字段 | 类型 | 来源与说明 |
|------|------|------------|
| `id` | string | 三位数字，与文件名编号一致，如 `"151"` |
| `title` | string | 从出题建议中的标题取（如「加法计算器」「加减法口算（期末真题）」），简短中文 |
| `description` | string | 题目描述，支持 Markdown。从出题建议的「描述」「GOC 改编」等合并；含输入输出说明、示例（可写「示例：输入 `12 25` → 输出 `37`」） |
| `difficulty` | string | `"easy"` / `"medium"` / `"hard"`，与出题建议一致 |
| `tags` | array[string] | 从出题建议的 tags 表取，如 `["cin输入","计算输出","加法"]` |
| `starterCode` | string | 可为 `"// 在此输入代码\n"` 或空字符串 `""`，或根据题目给一句注释 |
| `solution` | string | 出题建议中的完整 GOC 代码（C++ 风格），需符合 01 指令规范，可直接运行 |

### 4.2 可选字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `hint` | string | 有则从出题建议提取，否则可省略或空字符串 |
| `constraints` | string | 有则写（如「必须使用 for 循环」），否则可省略 |
| `expectedTrajectory` | array | **图形题**（solution 中含 `pen.`）：可先写 `[]`，步骤六统一跑 `gen-trajectories.js` 写入；**纯 cout 题**：保持 `[]` |
| `defaultInput` | string | **含 `cin` 的题目**：若有示例输入，写于此（如 `"12 25"`、`"5"`、多行用 `\n`），供 OJ 预览目标图形或将来输出判题；无则省略 |

### 4.3 文件名与题目名

- 文件名：`编号-题目名.json`，题目名 4～8 字中文，与 `title` 一致或高度一致。
- 若题目名含括号等符号，文件名中可简化（如 `151-加法计算器.json`）。

## 五、执行步骤（按序执行）

### 步骤 1：确认当前题库最大编号

- 查看 `problems/index.json` 或 `problems/` 下已存在的 `XXX-*.json`，确认最大编号为 150。
- 若你本地最大编号不是 150，则以你看到的**最大编号**为基准，下一批从 **最大编号+1** 起。

### 步骤 2：生成第一批 60 题（P-001～P-060 → 151～210）

- 打开 `docs/出题建议-按大纲学段.md`，按顺序处理每个 `### P-XXX` 小节。
- 对每个 P-XXX：生成一个 JSON 文件，编号 = 150 + P 的序号（P-001→151，P-002→152，…，P-060→210）。
- 从各小节表格与描述中提取：title、年级/知识点（可并入 description）、难度、tags、描述、示例、solution；无输入则不必写 defaultInput；有输入则从示例中取 defaultInput。
- 图形题（输出模式为「图」或含 `pen.`）：`expectedTrajectory` 先设为 `[]`。
- 计算题（仅 cout）：`expectedTrajectory` 为 `[]`。

### 步骤 3：生成第二批题（T-01～T-48，去重后 211 起）

- 打开 `docs/出题建议-真题改编.md`，按顺序处理每个 `### T-XX` 小节。
- 对每个 T-XX：判断是否与**已有题目**（001～150 及 151～210）题意与输入输出**实质相同**。例如：T-01 加减法 与 P-001/P-002 合并后的「和与差」若已有，可视为重复；T-03 乘法口诀表 与 P-003 可能重复。
- 若**不重复**：生成新题 JSON，编号从 211 起连续分配。
- 若**重复**：跳过该 T-XX，不生成文件。
- 新题同样遵守 07 Schema；含 cin 且有示例输入时写 `defaultInput`。

### 步骤 4：更新 index.json

- 打开 `problems/index.json`，在数组末尾**按编号顺序**追加所有新题条目。
- 每条格式与现有一致，例如：`{ "id": "151", "title": "加法计算器", "difficulty": "easy", "tags": ["cin输入","计算输出","加法"], "file": "151-加法计算器.json" }`。

### 步骤 5：运行 sync 脚本

- 在 `goc/` 目录下执行：`node scripts/sync-problems.js`（或项目内约定的同步命令）。
- 确认生成或更新了 `oj/data/problems.js`，且新题均出现在 `window.PROBLEMS_DATA` 中。

### 步骤 6：图形题 expectedTrajectory 预生成（可选但建议）

- 对 `expectedTrajectory` 为 `[]` 且 solution 中**不含 cin** 的题目，运行：`node scripts/gen-trajectories.js`（若脚本支持「只处理空轨迹」）。
- 若脚本会跳过含 cin 的题，则只处理纯图形题；含 cin 的图形题保持 `[]`，或根据项目约定后续用 defaultInput 在 OJ 内动态生成。
- 若本地无 Node 或脚本不可用，可保留 `[]`，在实现记录中注明「待后续批量生成轨迹」。

### 步骤 7：更新实现记录并汇报

- 编辑 `goc/docs/实现记录.md`，新增一节「按出题建议（大纲学段+真题改编）出题」，记录：
  - 新增题目数量：第一批 60 题（151～210），第二批 X 题（211～???）；
  - 按学段/题型的简要分布（可与出题建议中的概览表对齐）；
  - 涉及的脚本与文件（index.json、sync、gen-trajectories 等）。
- 向用户汇报：共新增多少题、编号区间、是否已运行 sync 与轨迹生成。
