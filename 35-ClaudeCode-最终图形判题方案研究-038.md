## 五、研究文档必须包含的内容

请在 `goc/docs/最终图形判题-方案研究.md` 中至少包含以下结构（可增补小节）：

### 5.1 问题与目标

- 简述当前判题方式及其局限（不同思路画同图判错）。
- 目标：支持「按最终图形等价」通过，或给出可落地的折中方案。

### 5.2 可行方案（至少三种）

对每种方案给出：

- **思路**：如何得到「参考图形」与「用户图形」，如何比较。
- **优点 / 缺点**：判题准确性、对「多解」的包容度、误判风险（如抗锯齿、浮点差异导致像素不完全一致）。
- **与现有架构的兼容性**：是否仍用现有 Canvas 与 `GOCRenderer.render`；是否需要后端、是否需要预存参考图或哈希。
- **实现难度与性能**：大致工作量；在浏览器内对典型题目（如 500×500 画布、数百步轨迹）的耗时与内存是否可接受。

建议覆盖的方向（可合并或拆分为至少三种方案）：

1. **像素/ImageData 比对**：在固定尺寸与 fit 下分别渲染 expectedTrajectory 与用户 trajectory，用 `ctx.getImageData` 取像素，逐像素或分块比较，允许少量像素差（容差）。
2. **图像哈希（如 pHash、dHash 或简单灰度+缩放哈希）**：对渲染结果算哈希，用户图与参考图哈希距离小于阈值则通过；对缩放/轻微位移的鲁棒性。
3. **多组 accepted 轨迹**：题目除 expectedTrajectory 外可配置多组「也视为正确」的轨迹（如预生成几种常见写法），判题时用户轨迹与任一组匹配即通过；仍为轨迹比对，但允许多解。
4. **轨迹规范化/等价类**：在保持「轨迹比对」的前提下，对轨迹做规范化（如合并连续同向 fd、统一起点/朝向）使等价画法归为同一序列再比；实现难度与边界情况需说明。
5. **其他**：如轮廓提取后比对、与现有方案组合等，可自选补充。

### 5.3 推荐方案及理由

- 明确推荐一种（或组合）方案，并写清理由（兼顾教学需求、实现成本、可维护性、与无后端/纯前端的兼容性）。

### 5.4 若采纳推荐方案：实现要点与数据/接口变更

- **接口**：`GOCJudge` 是否保留 `judge(userTraj, expectedTraj)`、是否新增如 `judgeByImage(userTraj, expectedTraj, canvasW, canvasH)` 或 `judgeByHash(...)` 等；调用方（如 index.html 提交逻辑）如何选择判题方式（如按题目配置或题型）。
- **数据**：题目 JSON 是否需要新字段（如 `judgeMode: "trajectory"|"image"`、`expectedImageHash`、参考图 base64 等）；若用多组轨迹，是否扩展为 `acceptedTrajectories: []`。
- **实现要点清单**：分条列出（例如：1. 在固定尺寸与 fit 下渲染参考图并缓存 ImageData 或哈希；2. 提交时同样尺寸与 fit 渲染用户轨迹，取 ImageData/算哈希；3. 比对逻辑与容差/阈值；4. 题目 schema 与 sync 脚本的兼容），便于后续写「实现类」Claude Code 任务说明。
