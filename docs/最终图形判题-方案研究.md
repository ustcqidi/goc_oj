# 最终图形判题 — 方案研究（GOC-038）

> 产出日期：2026-03-18
> 对应任务说明：`35-ClaudeCode-最终图形判题方案研究-038.md`

---

## 一、问题与目标

### 1.1 现状

GOC OJ 当前图形题判题在 `oj/js/goc-executor.js` 的 `GOCJudge.judge(userTraj, expectedTraj)` 中实现：

```
判题 = 比对 用户轨迹 vs expectedTrajectory
规则：
  1. 步数相同（过滤掉 speed/wait/cout 等非绘图指令后）
  2. 每步 cmd 相同（字符串精确匹配）
  3. 每步 args 在容差内（距离 ±2，角度 ±1°，颜色/线宽精确匹配）
```

**调用链：**
```
用户点击「提交」
  → GOCExecutor.run(code) → userResult.trajectory
  → ensureExpected(p)      → expectedTrajCache[p.id]（来自 JSON 或运行 solution）
  → GOCJudge.judge(userResult.trajectory, expected)
  → 展示 pass / fail + reason
```

### 1.2 核心问题

**不同思路画出视觉相同的图形，会被判错。**

典型场景：

| 场景 | 举例 |
|------|------|
| 起始方向不同 | solution 先画横边，学生先画竖边 |
| 正方形不同朝向 | `rt(90)` 循环 vs `lt(90)` 循环 |
| 等价命令替换 | `fd(100).rt(90)` 换成 `fd(50).rt(0).fd(50).rt(90)` |
| 多边形方向相反 | 顺时针 vs 逆时针（视觉镜像相同，轨迹完全不同） |
| moveTo 路径不同 | 先画上边再画下边 vs 先画左边再画右边 |
| cin 题多解 | 循环顺序相同但颜色读入顺序不同 |

这与 GOC 教学预期（"只要画出来的图对就行"）产生冲突，也会让学生困惑（"我画的和参考答案一模一样，为什么不通过？"）。

### 1.3 目标

1. **主目标**：支持「视觉等价」的多种正确写法都能通过，降低学生因笔画顺序不同而被误判的比例。
2. **约束**：纯前端（无后端）、浏览器 Canvas 2D、保持现有 `GOCRenderer.render` 可用、不引入外部服务。
3. **可接受的折中**：对部分题型保留轨迹严格比对（如有明确步骤要求的题），对其余题型改为图形比对。

## 二、可行方案

### 方案 A：像素 / ImageData 比对

#### 思路

在固定尺寸（如 400×400）和相同的 fitMode 下，将 **expectedTrajectory** 和 **用户轨迹** 分别渲染到两个离屏 Canvas，然后用 `ctx.getImageData()` 取出所有像素的 RGBA 数组，逐像素（或分块采样）比较差异，允许一定比例的像素不一致（如 < 2%）。

**判题函数签名（设计）：**

```javascript
// 新增判题方式
function judgeByImage(userTraj, refTraj, canvasW, canvasH, fitMode) {
  const refCanvas = createOffscreen(canvasW, canvasH);
  const userCanvas = createOffscreen(canvasW, canvasH);
  GOCRenderer.render(refCanvas.ctx,  refTraj,  canvasW, canvasH, fitMode);
  GOCRenderer.render(userCanvas.ctx, userTraj, canvasW, canvasH, fitMode);

  const refData  = refCanvas.ctx.getImageData(0, 0, canvasW, canvasH).data;
  const userData = userCanvas.ctx.getImageData(0, 0, canvasW, canvasH).data;

  let diffPixels = 0;
  for (let i = 0; i < refData.length; i += 4) {
    const dr = Math.abs(refData[i]   - userData[i]);
    const dg = Math.abs(refData[i+1] - userData[i+1]);
    const db = Math.abs(refData[i+2] - userData[i+2]);
    if (dr + dg + db > threshold) diffPixels++;
  }
  const totalPixels = canvasW * canvasH;
  return { pass: diffPixels / totalPixels < pixelTolRatio };
}
```

#### 优点

- **最直接**：真正比对「看到的结果是否一样」，与教学预期完全一致
- **天然容忍多解**：任何画出相同图形的代码都通过，不管笔画顺序
- **现有架构高度兼容**：`GOCRenderer.render` 已可用，只需创建离屏 Canvas，浏览器原生支持 `OffscreenCanvas` 或 `createElement('canvas')`
- **无需修改题目数据**：只需 expectedTrajectory（渲染成图）而非额外存储

#### 缺点

- **抗锯齿导致的像素差异**：Canvas 2D 绘制线段时有亚像素渲染，相同图形在不同渲染顺序下可能产生 1-3 像素的细微差别（尤其斜线、圆弧边缘），需要合理设置容差
- **fitMode 影响**：若用户图和参考图的 bounding box 不同，fitMode 下缩放比例不同，会导致像素位置偏移；需统一用相同 fitMode（且最好固定 bounding box 来源）
- **性能**：400×400 = 160,000 像素 × 4 通道 = 640KB 数据，遍历约需 1-3ms（现代浏览器可接受）；若 canvas 更大（800×800）则 ~10ms
- **颜色容差**：填充圆（oo）和描边圆（o）颜色差异明显，但抗锯齿边缘像素为混合色，需允许颜色误差（如 RGB 各分量差 ≤ 30）
- **对比例缩放不敏感**：若题目要求"半径 50"但学生画"半径 48"，像素差异可能超过阈值，也可能刚好在容差内，边界模糊
- **错误反馈弱**：只能说「图形不匹配」，无法指出具体哪里错了

#### 架构兼容性

- 浏览器端：直接创建 `document.createElement('canvas')` 离屏渲染，无需后端
- 可复用现有 `GOCRenderer.render(ctx, traj, w, h, fitMode)` 不做修改
- `goc-executor.js` 内无法直接使用（无 DOM），需在 `index.html` 的判题逻辑中调用

#### 实现难度

- **工作量**：约 30-50 行新增 JS（判题函数 + offscreen canvas 创建）
- **主要难点**：容差调参（像素比例阈值、颜色分量阈值），以及 fitMode 一致性问题
- **POC 验证点**：同一图形用两种顺序画，像素差异有多大？（需实测）

### 方案 B：图像感知哈希（pHash / dHash）

#### 思路

将两张渲染结果缩放为小图（如 16×16 或 32×32 灰度图），计算感知哈希（差值哈希 dHash 或均值哈希 aHash），比较两个哈希的 Hamming 距离，距离小于阈值（如 ≤ 10）则通过。

**dHash 简化实现（16×16 缩放后比较相邻像素）：**

```javascript
function dHash(imageData, w, h) {
  // 1. 缩放到 9×8 灰度图（dHash 需要 n+1 宽度）
  // 2. 比较每行相邻像素：左像素 > 右像素 → 1，否则 → 0
  // 3. 得到 64 位哈希字符串
}

function hammingDistance(h1, h2) {
  let dist = 0;
  for (let i = 0; i < h1.length; i++) if (h1[i] !== h2[i]) dist++;
  return dist;
}
```

#### 优点

- **比较快**：哈希计算 + Hamming 距离比全像素遍历更快（特别是复用缓存哈希时）
- **可缓存**：题目的参考图哈希可在首次渲染后缓存（甚至预存在题目 JSON 的 `expectedImageHash` 字段），后续提交只需算用户图哈希
- **对轻微位置偏移有鲁棒性**：pHash/aHash 对小幅移动和缩放不敏感，适合 fitMode 下比较
- **判题标准客观**：Hamming 距离 ≤ N 作为阈值，语义清晰

#### 缺点

- **纯 JS 实现图像缩放麻烦**：Canvas 2D 可以用 `drawImage` 缩放到小 canvas，但需要额外的 canvas 元素；灰度转换需逐像素手算
- **对 GOC 场景效果存疑**：GOC 图形多为细线条（1-3px）在大画布上，缩放到 16×16 后线条可能完全不可见，哈希失去区分能力；GOC 图形密度低，缩略图信息极少
- **阈值难定**：不同题目的"相似度"差异很大，无法用统一阈值，每题可能需要单独调参
- **错误反馈极弱**：只有"相似度不足"，对学生完全不透明
- **引入算法依赖**：需从零实现感知哈希，或引入第三方库（与纯前端无外部依赖的架构有冲突）

#### 架构兼容性

- 可在浏览器端实现（无需后端），但需要新增图像处理逻辑
- 若预存哈希，需修改题目 JSON 增加 `expectedImageHash` 字段，并修改 sync 脚本在生成轨迹时同步生成哈希
- 在 Node 环境（gen-trajectories.js）生成哈希时，Node 无原生 Canvas，需要 `canvas` npm 包

#### 实现难度

- **工作量**：约 80-120 行（哈希算法 + 缩放处理 + 阈值调参 + JSON 字段扩展 + sync 脚本更新）
- **主要难点**：GOC 细线图形缩略哈希的有效性验证、统一阈值的不可行性
- **结论**：对 GOC 细线图形场景，感知哈希效果可能不如方案 A，且实现更复杂，**不推荐作为主方案**

### 方案 C：多组 Accepted 轨迹（acceptedTrajectories）

#### 思路

在题目 JSON 中增加 `acceptedTrajectories: [traj1, traj2, ...]` 字段，包含多种正确写法的轨迹（如顺时针/逆时针各一种，或若干典型解法）。判题时，用户轨迹与**任意一组** accepted 轨迹匹配（按现有严格比对）即通过。

**判题逻辑扩展：**

```javascript
function judgeMulti(userTraj, problem) {
  const allExpected = [
    ...(problem.expectedTrajectory ? [problem.expectedTrajectory] : []),
    ...(problem.acceptedTrajectories || [])
  ];
  for (const expected of allExpected) {
    const verdict = GOCJudge.judge(userTraj, expected);
    if (verdict.pass) return { pass: true, reason: '通过！' };
  }
  return { pass: false, reason: '未找到与任何参考答案匹配的写法。' };
}
```

#### 优点

- **改动最小**：不修改 `GOCJudge.judge` 核心逻辑，不涉及 Canvas 渲染，浏览器/Node 双端兼容
- **错误反馈保留**：对单组匹配失败仍可给出"第 N 步不同"的具体提示（可改为"所有参考写法均不匹配"）
- **精准控制**：出题者可明确定义"哪些写法视为正确"，比图像比对更可控
- **性能极佳**：纯数组遍历，与现有判题性能一致
- **渐进式**：与现有 `expectedTrajectory` 完全向后兼容

#### 缺点

- **扩展性差**：穷举所有可能的正确写法是不现实的；一道题可能有几十种等效写法
- **维护成本高**：每次出新题都需要手动生成若干种 accepted 轨迹，出题工作量增加 2-3 倍
- **本质不变**：仍是轨迹比对，仍有「漏掉的写法」被错判的风险
- **数据量增大**：每道题多存若干组轨迹（每组可能 50-500 步），JSON 文件体积增加明显

#### 架构兼容性

- **最高**：只需扩展 JSON 字段、修改判题调用处（index.html），其余不变
- sync 脚本和 gen-trajectories.js 只需少量改动（支持读写 acceptedTrajectories）

#### 实现难度

- **工作量（代码层面）**：约 10-20 行（判题逻辑扩展 + index.html 调用修改）
- **工作量（内容层面）**：大——每道图形题需额外提供 1-3 种常见写法的轨迹
- **结论**：作为**过渡/补充方案**可行，不适合作为长期主方案

### 方案 D：轨迹规范化 / 等价类归一

#### 思路

在保持轨迹比对的前提下，对用户轨迹和参考轨迹各自执行**规范化变换**，使等价画法归为同一序列，再比对规范化后的轨迹。

典型的规范化规则（举例）：

1. **合并连续同向移动**：`fd(50).fd(50)` → `fd(100)`
2. **合并连续同向旋转**：`rt(30).rt(60)` → `rt(90)`；`rt(360)` → 消除
3. **统一绝对角度**：记录「当前绝对朝向」而非相对旋转，比较最终像素路径而非旋转命令序列
4. **起点/朝向归一化**：将画布中第一条有效线段的起点平移至原点、朝向归零，之后所有坐标相对变换
5. **up/down 对消**：`up().moveTo(x,y).down()` 的路径信息只保留最终 moveTo 目标

#### 优点

- **纯轨迹操作**：无需 Canvas 渲染，在 Node 端（gen-trajectories.js）也可用
- **错误反馈清晰**：规范化后仍可说明"第 N 步不同"
- **对简单等价关系效果好**：连续 fd 合并、旋转合并等场景完全覆盖

#### 缺点

- **边界情况极多**：GOC 语言允许 pen.up/down、moveTo、圆弧等非线性路径，规范化规则覆盖完全极难；每新增一条规则都可能引入新的歧义
- **实现复杂度高**：需重新模拟执行过程（记录绝对坐标和状态），等价于实现一个新的「语义等价分析器」
- **仍有根本局限**：即使规范化后，顺时针 vs 逆时针画的正方形（坐标相同、方向相反）轨迹仍然不同，规范化难以捕捉所有视觉等价关系
- **维护风险**：规则集随语言扩展需同步维护，否则新指令破坏规范化

#### 架构兼容性

- 可在 `goc-executor.js` 内新增 `normalizeTraj(traj)` 函数，Node/浏览器双端兼容
- 不需修改 JSON Schema 或渲染逻辑

#### 实现难度

- **工作量**：约 150-300 行（规范化函数 + 单元测试覆盖各种边界情况）
- **主要难点**：边界情况的全面覆盖几乎不可能；规范化失败时的 fallback 策略
- **结论**：作为对方案 A 的**补充优化**可考虑（先规范化再图像比对），单独作为主方案风险较高

### 方案 E：分层判题（轨迹比对 + 图像兜底）

#### 思路

结合方案 A 和现有轨迹判题，形成两级判题：

```
第一级：GOCJudge.judge(userTraj, expected)
  ↓ pass → 直接通过，给出「与标准答案一致」
  ↓ fail → 进入第二级

第二级：judgeByImage(userTraj, refTraj, w, h, fitMode)
  ↓ pass → 通过，给出「图形正确，但写法与参考答案不同」
  ↓ fail → 不通过，给出「图形不匹配」
```

对题目设置 `judgeMode` 字段控制是否启用第二级：
- `"trajectory"`：仅轨迹比对（用于明确要求步骤的练习）
- `"image"`：仅图像比对（最宽松）
- `"image_fallback"`（默认）：轨迹匹配优先，图像兜底

## 四、推荐方案及理由

### 推荐：方案 E（分层判题 = 轨迹比对优先 + 像素图像兜底）

#### 推荐理由

1. **完全向后兼容**：第一级保留现有 `GOCJudge.judge`，已入库的 150+ 题、现有通过记录全部不受影响。

2. **教学预期匹配**：图像兜底真正解决「明明画对了却判错」的问题，与 GOC 教学理念（画出正确图形即可）一致。

3. **纯前端可实现**：只需在 `index.html` 的提交逻辑中增加 `<canvas>` 离屏渲染 + `getImageData` 比对，无需后端或外部依赖，约 40-60 行新增代码。

4. **错误反馈分级**：
   - 轨迹通过 → "与标准答案写法完全一致 ✅"
   - 图像通过 → "图形正确！你的写法与参考答案不同，但结果一样 ✅"
   - 两级均不通过 → "图形不匹配，请检查你的代码 ❌"（后续可增加点击查看参考图的功能）

5. **可控性强**：通过 `judgeMode` 字段，出题者可对特定题型（如"必须按特定步骤"的练习题）仍用纯轨迹判题。

6. **实现路径清晰**：两个阶段可独立测试，先上第一级（已有），再加第二级图像判题，风险可控。

#### 次要推荐：方案 C（多组轨迹）作为短期过渡

在方案 E 实现完成之前，可先为高价值题目（如正方形、等边三角形等有明显多解的题）手动添加 1-2 组额外 accepted 轨迹。实现成本极低（10 行代码），可立即缓解最严重的误判问题。

## 五、推荐方案实现要点与数据/接口变更

### 5.1 题目数据（JSON Schema）变更

#### 新增可选字段

```json
{
  "judgeMode": "image_fallback",
  "acceptedTrajectories": [],
  "expectedImageHash": ""
}
```

| 字段 | 类型 | 默认值 | 说明 |
|------|------|--------|------|
| `judgeMode` | string | `"image_fallback"` | `"trajectory"` / `"image"` / `"image_fallback"` |
| `acceptedTrajectories` | array | `[]` | 额外接受的轨迹组（方案 C 兼容） |
| `expectedImageHash` | string | `""` | 参考图的简单哈希（可选，用于快速预检；留空则实时渲染比对） |

**向后兼容**：三个字段均为可选，旧题不修改也能正常工作（`judgeMode` 未填时默认 `"image_fallback"`，或保留现有行为用 `"trajectory"` 作为默认值更安全）。

> **建议默认值策略**：初期默认 `"trajectory"`（保持现有行为），在测试稳定后逐步改为 `"image_fallback"`。

### 5.2 接口变更

#### `GOCJudge` 扩展（goc-executor.js）

```javascript
// 新增：不修改现有 judge，新增 judgeByImage
global.GOCJudge = {
  judge,           // 现有：轨迹比对
  judgeByImage,    // 新增：像素比对（需在有 DOM 的环境调用）
  judgeAuto,       // 新增：根据 judgeMode 自动选择
};
```

`judgeByImage` 需要 DOM（Canvas），因此不能直接放入 goc-executor.js（Node 端无 DOM），应放在 `index.html` 的 `<script>` 块或单独的 `goc-judge-image.js`。

**函数签名设计：**

```javascript
// 在 index.html 或 goc-judge-image.js 中
function judgeByImage(userTraj, refTraj, options) {
  // options: { canvasW, canvasH, fitMode, pixelTolRatio, colorTol }
  // 默认: canvasW=400, canvasH=400, fitMode=true, pixelTolRatio=0.02, colorTol=30

  const refCtx  = createOffscreenCtx(options.canvasW, options.canvasH);
  const userCtx = createOffscreenCtx(options.canvasW, options.canvasH);

  GOCRenderer.render(refCtx,  refTraj,  options.canvasW, options.canvasH, options.fitMode);
  GOCRenderer.render(userCtx, userTraj, options.canvasW, options.canvasH, options.fitMode);

  return compareImageData(
    refCtx.getImageData(0, 0, options.canvasW, options.canvasH),
    userCtx.getImageData(0, 0, options.canvasW, options.canvasH),
    options.pixelTolRatio,
    options.colorTol
  );
}
```

#### `index.html` 提交逻辑修改

**当前（submitCode 中约 1291 行）：**
```javascript
const verdict = GOCJudge.judge(userResult.trajectory, expected);
if (verdict.pass) { ... } else { ... }
```

**修改为：**
```javascript
const judgeMode = currentProblem.judgeMode || 'trajectory'; // 初期保守默认

let verdict;
if (judgeMode === 'trajectory') {
  // 纯轨迹判题（保持现有）
  verdict = judgeMulti(userResult.trajectory, currentProblem, expected);
} else if (judgeMode === 'image') {
  // 纯图像判题
  verdict = judgeByImage(userResult.trajectory, expected, IMAGE_JUDGE_OPTIONS);
} else {
  // image_fallback：先轨迹，再图像
  verdict = judgeMulti(userResult.trajectory, currentProblem, expected);
  if (!verdict.pass) {
    const imgVerdict = judgeByImage(userResult.trajectory, expected, IMAGE_JUDGE_OPTIONS);
    if (imgVerdict.pass) {
      verdict = { pass: true, reason: '图形正确！你的写法与参考答案不同，但画出的图形一样。' };
    }
  }
}
```

其中 `judgeMulti` 是对方案 C 的兼容处理（检查 acceptedTrajectories）：

```javascript
function judgeMulti(userTraj, problem, primaryExpected) {
  const allExpected = [primaryExpected, ...(problem.acceptedTrajectories || [])];
  for (const exp of allExpected) {
    const v = GOCJudge.judge(userTraj, exp);
    if (v.pass) return v;
  }
  // 返回第一组的失败信息
  return GOCJudge.judge(userTraj, primaryExpected);
}
```

### 5.3 图像比对关键参数

| 参数 | 推荐初始值 | 说明 |
|------|-----------|------|
| `canvasW / canvasH` | 400×400 | 固定尺寸，比 OJ 显示尺寸小（更快），足以区分大多数图形 |
| `fitMode` | `true` | 参考图和用户图都用 fitMode，确保缩放一致 |
| `pixelTolRatio` | 0.02（2%） | 允许 2% 的像素不一致，容纳抗锯齿差异 |
| `colorTol` | 20（RGB 各分量） | 单通道差值 ≤ 20 视为"颜色一致" |

> **需要 POC 验证**：对同一图形用两种写法渲染后，典型的像素差异比例是多少？建议在固定参数下实测 3-5 个典型题目（正方形、等边三角形、同心圆），确认 2% 容差是否合适。

### 5.4 fitMode 一致性问题（关键细节）

`computeFitTransform` 依赖 **bounding box**，由 trajectory 内容决定。若用户写法的 bounding box 和参考答案不同（如用户多画了一步 bk 超出范围），fitMode 下的缩放比例不同，会导致像素偏移。

**解决方案**：判题时统一使用**参考轨迹的 fitTransform** 渲染两张图，而非各自 fit：

```javascript
// 使用参考轨迹计算 transform，两张图都用同一 transform
const refTransform = GOCRenderer.computeFitTransform(refTraj, w, h, 20);
GOCRenderer.renderWithTransform(refCtx, refTraj, w, h, refTransform);
GOCRenderer.renderWithTransform(userCtx, userTraj, w, h, refTransform);
```

这需要 `GOCRenderer` 新增一个接受外部 transform 的渲染方法（或复用内部 `applyStep`），是需要修改 `goc-executor.js` 的唯一一处。
