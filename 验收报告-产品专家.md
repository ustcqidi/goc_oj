## 二、产品逻辑与体验核验

### 2.1 判题链路

- **预期**：用户写代码 → 运行看图 → 提交 → 与参考答案轨迹比对 → 显示通过/未通过。
- **实现**：OJ 未使用 JSON 中预填的 `expectedTrajectory`（当前全为空数组），而是在**打开题目时**用执行器跑一遍 `solution`，将得到的轨迹缓存为「期望轨迹」；提交时再跑用户代码得到轨迹，与缓存比对。这样无需在题目 JSON 中预生成轨迹，与「无后端、静态部署」的 MVP 一致，判题逻辑有效。
- **评价**：✅ 设计合理，符合 MVP 目标；若未来需要离线预填轨迹（如 CDN 只托管 JSON），可再补生成脚本。

### 2.2 题目与难度分布

- **数量**：15 道，覆盖 001～015。
- **难度**：入门（easy）约 8 道、进阶（medium）约 5 道、高级（hard）2 道，以入门为主，符合计划「10～20 道、入门为主」。
- **题型**：正多边形、五角星、同心圆、楼梯、放射线、螺旋、风车、渐大正方形、方格网、奇偶双色、彩虹螺旋等，标签与 08 分类一致，题干含绘图要求与数学点，无重复感。
- **评价**：✅ 题目质量与分布符合 06/08/09 规范，可直接用于孩子练习。

### 2.3 文档一致性

- 07 与 10：07 中 `expectedTrajectory` 格式说明与 10 一致，并引用 10。
- 08 与题目：题目中 difficulty、tags 取值均在 08 定义的范围内。
- 01 与执行器：执行器支持的指令为 01 子集，轨迹 cmd 与 10 表一致。
- **评价**：✅ 文档与实现一致，无冲突。

## 三、改进建议（非阻塞）

| 建议 | 说明 |
|------|------|
| **题目数据与 OJ 数据同步** | 当前 OJ 从 `oj/data/problems.js` 内嵌数据加载题目，而题目源是 `problems/*.json`。新增或修改题目时需同步更新 `problems.js`，否则 OJ 看不到新题。建议：增加一条「从 problems/*.json 生成 problems.js」的脚本或说明，或改为前端 fetch 静态 JSON（需考虑 file:// 跨域），避免双源维护。 |
| **07 / 交付清单对 expectedTrajectory 的表述** | 交付清单写「每题含 expectedTrajectory」；当前实现为「可为空，由 OJ 运行时用 solution 生成」。建议在 07 或 05 中明确写一句：「OJ MVP 下 expectedTrajectory 可留空，由前端打开题目时执行 solution 动态生成」，与实现一致，避免后续阅读歧义。 |
| **首题端到端自检** | 建议在 README 或 05 中注明「验收时用 001 画正方形走通：选题 → 运行 → 提交正确代码 → 显示通过；改错代码 → 提交 → 显示未通过」，便于以后回归验证。 |
